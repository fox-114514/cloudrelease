# StudyShot Relay 完整方案规格

文档版本：2026-06-17  
目标读者：后续负责编码实现的 AI、开发者、测试者  
项目暂名：StudyShot Relay，中文可称“学习截图快传”

## 0. 一句话目标

做一个私有的低延迟截图/图片快传系统：用户在安卓平板上写笔记并截图后，图片自动上传到自己的服务器；授权的电脑或手机自动接收并下载，最好自动放入剪贴板，让用户停留在 其他页面内直接粘贴或上传图片。

本项目不是网盘，不是完整相册同步，不是聊天软件，也不是 GPT API 客户端。

## 1. 背景与问题

用户拥有以下设备：

- 一台双系统笔记本：Windows 用于娱乐，Ubuntu 用于写代码和学习。
- 一台安卓手机：日常使用。
- 一台安卓平板：主要用于写作业、手写笔记。

学习流程：

1. 用户在平板上手写笔记、推导过程、作业答案。
2. 用户在电脑或手机上使用 GPT 辅助学习。
3. 用户需要把平板上的手写内容截图发给 GPT。

已尝试方案及问题：

- 平板分屏：左边 GPT，右边笔记软件。
  - 优点：交互直观。
  - 问题：平板发烫、变卡；分屏、GPT、笔记软件叠加后影响书写体验。
- 微信文件传输助手：
  - 优点：可用。
  - 问题：步骤多。平板截图、切微信、发送；电脑切微信、下载；再切 GPT、上传。
- 一加/OPPO 系自带中转站：
  - 优点：同品牌设备集成。
  - 问题：走厂商云端同步，延迟常常达到几十秒到一分钟，不适合高频学习场景。

核心问题不是“如何传文件”，而是如何降低跨设备传图的摩擦，并保持学习思路连续。

## 2. 产品定位

StudyShot Relay 是一个“私有低延迟图片中继系统”。

它解决的是：

- 安卓设备新增截图后自动上传。
- 授权设备自动接收。
- 电脑端自动下载到固定目录，最好自动写入剪贴板。
- 支持多用户、多设备、设备级权限。
- 避免上传设备把自己上传的图片再下载回来。
- 避免下载后的图片被再次监听并上传，造成循环。

它不解决：

- 不做完整网盘。
- 不做完整相册备份。
- 不做多人聊天。
- 不做图片编辑器。
- 不默认接入 GPT/OpenAI API。
- 不要求发布到应用商店。
- 不依赖一加、OPPO、小米、华为等厂商云服务。

## 3. 成功标准

### 3.1 体验目标

正常网络条件下，从平板截图完成到电脑收到图片：

- 理想：1 到 3 秒。
- 可接受：5 秒内。
- 弱网或服务器在境外时：10 秒内可接受，但客户端要显示状态。

电脑端收到后：

- 自动保存到固定目录。
- 可选自动复制最新图片到系统剪贴板。
- 用户停留在 GPT 页面时，可以直接 Ctrl+V 粘贴图片。

安卓端：

- 不明显影响手写笔记软件性能。
- 不持续扫描整个相册。
- 不需要用户每次切换到传输 App。
- 实时模式开启时可以有常驻通知；这是可接受的。

### 3.2 稳定性目标

- 截图监听不能漏掉常规截图。
- 同一张图片不能因系统多次通知而重复上传。
- 上传失败要自动重试。
- 设备离线期间收到的图片，重新上线后按规则补收。
- 服务器重启后客户端自动重连。
- 被撤销的设备不能继续上传、下载或接收事件。

### 3.3 安全目标

- 所有外网通信必须使用 HTTPS/WSS。
- 设备必须绑定后才能使用。
- 主用户可以撤销任意子用户和设备。
- 权限以设备为最小执行单位。
- 默认不上传整个相册，只上传截图或用户指定来源。
- 服务端需要保留审计日志，便于排查误传。

## 4. 角色、用户、设备模型

### 4.1 概念定义

主用户：

- 一个空间的拥有者。
- 可以创建子用户、用户组、设备绑定码。
- 可以管理所有用户和设备权限。
- 可以配置服务器级保留策略。

子用户：

- 归属于某个主用户。
- 可以拥有自己的设备。
- 权限由主用户授予。
- 可以被加入一个或多个用户组。

用户组：

- 主用户创建的用户集合。
- 用于批量配置权限。
- 例子：`我的设备`、`家人设备`、`临时协作`、`只上传组`。

设备：

- 权限执行的最小单位。
- 每台手机、平板、电脑都是一个独立设备。
- 设备属于某个用户，同时也归属于对应主用户的空间。
- 设备有独立 token，可以被单独撤销。

图片：

- 一次上传产生一个图片对象。
- 图片有上传设备、上传用户、所属主用户空间、哈希、存储位置、过期时间。

投递：

- 服务器根据权限为某张图片生成一个或多个目标设备投递记录。
- 投递记录有状态：待通知、已通知、已下载、失败、跳过、过期。

### 4.2 为什么权限必须落在设备上

不要只做“用户能上传/用户能下载”。

真实需求往往是：

- 平板可以自动上传，但不自动接收。
- Ubuntu 笔记本可以自动接收并写剪贴板。
- Windows 系统也可以接收，但只在娱乐时手动打开。
- 手机可以手动上传，也可以自动接收，但不一定要常驻后台。

因此必须支持设备级权限：

- 某设备能否自动上传。
- 某设备能否手动上传。
- 某设备能否自动接收。
- 某设备能否手动下载。
- 某设备可以接收哪些来源设备的图片。
- 某设备上传的图片可以投递给哪些设备。

### 4.3 主用户与子用户关系

每个空间必须有且只有一个主用户。

子用户必须归属于一个主用户空间。子用户不能跨主用户空间共享设备和图片。

推荐结构：

- `owner_user_id`：主用户 ID。
- `user_id`：实际操作用户 ID。
- 所有设备、图片、权限、组都带 `owner_user_id`，用于隔离空间。

### 4.4 绑定方式

不要用永久固定 ID 直接绑定。固定 ID 泄露后容易被冒充。

推荐绑定流程：

1. 新设备打开客户端。
2. 用户输入服务器地址。
3. 客户端生成本机设备 ID 和设备公钥或随机设备 secret。
4. 客户端显示一次性绑定码或二维码。
5. 主用户在已登录设备上确认绑定，或者子用户输入主用户生成的邀请码。
6. 服务器签发设备 token。
7. 后续请求使用设备 token。

绑定码要求：

- 一次性使用。
- 默认 10 分钟过期。
- 只能绑定到指定主用户空间。
- 使用后立即失效。
- 绑定时要记录设备名、平台、系统版本、App 版本、创建时间。

设备 token 要求：

- 长随机值或 JWT 加刷新机制均可。
- 服务端必须能单独撤销某台设备。
- token 不得明文存储在服务端；至少存 hash。
- 客户端本地要存入安全存储。
  - Android：EncryptedSharedPreferences 或 Jetpack Security。
  - Windows/Linux 桌面：优先系统凭据库；不可用时使用加密本地配置文件。

## 5. 权限模型

### 5.1 设备权限字段

每台设备至少要有以下权限：

- `can_auto_upload`：允许自动上传。
- `can_manual_upload`：允许手动上传。
- `can_auto_receive`：允许自动接收。
- `can_manual_download`：允许手动下载历史图片。
- `can_manage_space`：允许管理空间，仅主用户可信设备应开启。
- `can_create_invite`：允许创建用户或设备邀请。
- `is_revoked`：设备是否已撤销。

### 5.2 自动上传来源范围

每台设备需要配置自动上传范围：

- `screenshot_only`：默认值，只上传截图目录中的新图片。
- `selected_album`：用户选择的特定相册或目录。
- `manual_share_only`：不监听，只通过分享菜单或手动选择上传。
- `all_images`：不推荐，必须二次确认，并在界面上明确风险。

默认必须是 `screenshot_only`，不能默认监听整个相册。

### 5.3 接收范围

每台设备需要配置自动接收范围：

- 接收所有授权来源设备。
- 只接收同一用户的设备。
- 只接收指定用户组的设备。
- 只接收显式指定设备。
- 不自动接收，只允许手动查看。

推荐 MVP 支持：

- `all_authorized_sources`
- `same_user_only`
- `selected_devices`
- `disabled`

### 5.4 投递排除规则

必须硬性实现：

- 图片不会自动投递给上传设备本身。
- 如果目标设备没有 `can_auto_receive`，不生成自动投递。
- 如果来源设备没有 `can_auto_upload`，自动上传请求应返回 403。
- 如果设备被撤销，所有请求返回 401 或 403。
- 如果图片已经过期，不允许继续下载。

### 5.5 防循环规则

防循环不是一个可选优化，是核心需求。

必须同时在服务端和客户端做防护：

服务端：

- 每张图片记录 `upload_device_id`。
- 生成投递时排除 `upload_device_id`。
- 基于 `sha256` 做去重：同一空间、同一设备、短时间内相同 sha256 只保留一次。
- 如果客户端上传时带 `origin_image_id`，表示该图片来自服务器下载，服务端应拒绝自动上传或标记为循环风险。

安卓客户端：

- 自动下载的图片默认保存到 App 私有目录，不保存到系统相册。
- 如果用户选择保存到相册，必须默认不被自动上传；用本地哈希库跳过已下载图片。
- 自动上传前先计算 sha256，如果哈希存在于本机“已接收图片哈希表”，跳过。
- 下载目录必须放 `.nomedia`，避免被媒体库扫描。

桌面客户端：

- 自动下载目录不需要被上传监听，因为桌面端默认不做本地文件夹自动上传。
- 如果未来支持桌面自动上传，也必须默认不监听下载目录。

## 6. 整体架构

系统由三部分组成：

- 服务器后端。
- 安卓 App。
- 电脑客户端。

### 6.1 推荐通信方式

上传：

- 客户端通过 HTTPS multipart 或分片上传图片到服务器。

实时通知：

- 客户端与服务器保持 WebSocket 连接。
- 服务器收到新图片后，根据权限生成投递记录，并通过 WebSocket 通知目标设备。

下载：

- 目标设备收到事件后，通过 HTTPS 下载图片。
- 下载成功后向服务器 ACK。

离线补收：

- 客户端重连后调用“获取未完成投递”接口。
- 不依赖 WebSocket 消息永不丢失。

### 6.2 为什么不用轮询作为主方案

轮询可以作为降级方案，但不能作为主实时方案。

原因：

- 高频轮询耗电、耗流量。
- 低频轮询延迟高。
- 用户需求是截图后快速给 GPT 看，实时性很重要。

主方案必须是 WebSocket 或 SSE。桌面端和安卓端均可使用 WebSocket。

### 6.3 为什么不用厂商推送作为主方案

不要依赖 FCM、小米推送、OPPO 推送等作为主通道。

原因：

- 用户有自己的服务器。
- 安卓国内环境推送不稳定。
- 厂商推送适合通知，不适合确保图片立即下载。
- 实时学习流程要求客户端主动保持连接更可控。

可选：未来可以用推送作为“唤醒客户端”的辅助机制，但 MVP 不依赖它。

## 7. 推荐技术栈

### 7.1 后端

推荐：

- Node.js + TypeScript。
- Fastify 作为 HTTP 框架。
- WebSocket 使用 `ws` 或 Fastify WebSocket 插件。
- PostgreSQL 存储用户、设备、权限、图片元数据、投递记录。
- Prisma 或 Drizzle 作为数据库访问层。
- 本地磁盘或 S3 兼容对象存储保存图片。
- Docker Compose 部署。
- Caddy 或 Nginx 负责 HTTPS/WSS 反向代理。

原因：

- TypeScript 对后续 AI 编码友好。
- Fastify 简洁，适合 API 服务。
- PostgreSQL 足够稳定，权限和投递状态适合关系型数据库。
- 个人服务器部署简单。
- 1MB 级图片传输量不需要复杂分布式架构。

不要在 MVP 中引入：

- 微服务。
- Kafka。
- Kubernetes。
- GraphQL。
- 大型对象存储依赖。
- Firebase 作为核心后端。

### 7.2 安卓 App

推荐：

- Kotlin。
- Jetpack Compose 构建界面。
- OkHttp 负责 HTTP 和 WebSocket。
- WorkManager 负责可靠后台任务和失败重试。
- Foreground Service 负责实时学习模式。
- Room 存储本地队列、图片哈希、投递状态、设置。
- MediaStore + ContentResolver 读取新增图片。
- ContentObserver 监听媒体库变化。

最低支持：

- Android 10 及以上优先。
- 如果用户设备较新，可以以 Android 11/12+ 为实际测试重点。

权限策略：

- Android 13+ 使用 `READ_MEDIA_IMAGES`。
- Android 12 及以下使用 `READ_EXTERNAL_STORAGE`。
- Android 14+ 要处理“部分照片访问”情况，用户必须授权完整图片访问才能实现自动监听全部截图。
- 不要默认申请 `MANAGE_EXTERNAL_STORAGE`。除非用户明确接受，并且实现者说明原因。

相关官方资料：

- Android Photo Picker 与选择图片权限：https://developer.android.com/training/data-storage/shared/photo-picker
- Android 14 部分照片访问：https://developer.android.com/about/versions/14/changes/partial-photo-video-access
- ContentObserver：https://developer.android.com/reference/android/database/ContentObserver
- WorkManager Content URI 触发：https://developer.android.com/reference/androidx/work/Constraints.Builder#addContentUriTrigger(android.net.Uri,boolean)
- Foreground Service dataSync 类型：https://developer.android.com/develop/background-work/services/fgs/service-types

### 7.3 电脑客户端

推荐 MVP：

- Electron + TypeScript。
- 支持 Windows 与 Ubuntu。
- 使用系统托盘。
- 使用 WebSocket 接收事件。
- 使用 HTTPS 下载图片。
- 使用 Electron clipboard API 写入剪贴板。
- 使用本地 SQLite 或 JSON 配置保存设置和投递状态。

原因：

- 跨 Windows/Linux 成熟。
- 自动写剪贴板、托盘、开机自启、文件下载都容易实现。
- 后续 AI 编码更不容易卡在系统 API 上。

注意：

- Electron 资源占用比 Tauri 大，但本项目电脑端常驻开销仍可接受。
- 如果后续特别追求轻量，可以迁移到 Tauri 或原生 Rust/Go 客户端。
- Linux Wayland 下剪贴板图片支持可能因桌面环境不同而有差异。必须有降级策略：写入文件夹并弹出或记录“剪贴板写入失败”。

### 7.4 不推荐技术栈

不要把安卓端做成 WebView 套壳：

- 后台监听相册、权限、Foreground Service、WorkManager 都需要原生能力。

不要只做 PWA：

- PWA 无法可靠监听系统相册新增截图。

不要把桌面端只做网页：

- 网页无法稳定自动下载到固定目录，也无法可靠写系统剪贴板图片。

## 8. 后端详细需求

### 8.1 服务职责

后端负责：

- 用户登录。
- 主用户、子用户、用户组管理。
- 设备绑定和撤销。
- 设备权限配置。
- 图片上传。
- 图片存储。
- 图片元数据记录。
- 投递记录生成。
- WebSocket 实时通知。
- 图片下载鉴权。
- 下载 ACK。
- 去重。
- 过期清理。
- 审计日志。

后端不负责：

- OCR。
- GPT 调用。
- 图片内容理解。
- 图片编辑。
- 第三方云同步。

### 8.2 数据表建议

以下是逻辑模型，不要求一字不差，但语义必须保留。

#### users

- `id`：用户 ID。
- `owner_user_id`：所属主用户 ID。主用户自己的 `owner_user_id` 等于自己的 `id`。
- `role`：`owner` 或 `child`。
- `display_name`：显示名。
- `email_or_login`：登录名，可为空，取决于登录方案。
- `password_hash`：如果采用密码登录。
- `created_at`。
- `disabled_at`。

#### groups

- `id`。
- `owner_user_id`。
- `name`。
- `created_at`。

#### group_members

- `group_id`。
- `user_id`。
- `created_at`。

#### devices

- `id`：设备 ID，建议 UUID 或 ULID。
- `owner_user_id`。
- `user_id`：设备所属用户。
- `name`：用户自定义设备名，例如“一加平板”“Ubuntu 笔记本”。
- `platform`：`android`、`windows`、`linux`。
- `app_version`。
- `os_version`。
- `device_token_hash`。
- `public_key`：可选。
- `last_seen_at`。
- `created_at`。
- `revoked_at`。

#### device_permissions

- `device_id`。
- `can_auto_upload`。
- `can_manual_upload`。
- `can_auto_receive`。
- `can_manual_download`。
- `can_manage_space`。
- `can_create_invite`。
- `auto_upload_scope`：`screenshot_only`、`selected_album`、`manual_share_only`、`all_images`。
- `auto_receive_scope`：`disabled`、`all_authorized_sources`、`same_user_only`、`selected_devices`。
- `created_at`。
- `updated_at`。

#### receive_source_rules

用于 `selected_devices` 或更细粒度规则。

- `target_device_id`。
- `source_device_id`。
- `enabled`。
- `created_at`。

#### images

- `id`：图片 ID，建议 ULID，便于按时间排序。
- `owner_user_id`。
- `upload_user_id`。
- `upload_device_id`。
- `origin_image_id`：如果这张图来自某张已下载图片，记录原图 ID；正常截图为空。
- `sha256`。
- `mime_type`。
- `file_size`。
- `width`。
- `height`。
- `storage_key`。
- `source_kind`：`screenshot`、`manual_share`、`selected_album`、`unknown`。
- `source_display_name`。
- `source_media_id_hash`：安卓 MediaStore ID 不建议直接长期裸存，可存 hash。
- `created_at`。
- `expires_at`。
- `deleted_at`。

#### deliveries

- `id`。
- `image_id`。
- `target_device_id`。
- `status`：`pending`、`notified`、`downloaded`、`failed`、`skipped`、`expired`。
- `failure_reason`。
- `notified_at`。
- `downloaded_at`。
- `created_at`。
- `updated_at`。

#### bind_codes

- `id`。
- `owner_user_id`。
- `created_by_user_id`。
- `code_hash`。
- `purpose`：`bind_device`、`invite_child_user`。
- `target_role`：可选。
- `expires_at`。
- `used_at`。
- `used_by_device_id`。
- `created_at`。

#### audit_logs

- `id`。
- `owner_user_id`。
- `actor_user_id`。
- `actor_device_id`。
- `action`。
- `target_type`。
- `target_id`。
- `metadata_json`。
- `created_at`。

### 8.3 API 约定

API 路径统一使用 `/api/v1`。

所有需要鉴权的接口：

- 使用 `Authorization: Bearer <token>`。
- 服务端根据 token 找到设备和用户。
- 每次请求更新 `last_seen_at`。

#### 登录/初始化

`POST /api/v1/auth/login`

用途：

- 主用户或子用户登录已有账号。
- 登录成功后可以创建绑定码或注册新设备。

输入字段：

- `login`。
- `password`。

输出字段：

- `access_token`。
- `refresh_token`。
- `user`。
- `owner_user_id`。

MVP 可以采用更简单方案：

- 服务器首次启动时通过环境变量创建主用户。
- 之后主用户在客户端登录。

#### 创建设备绑定码

`POST /api/v1/bind-codes`

权限：

- `can_create_invite` 或 `can_manage_space`。

输入字段：

- `purpose`：`bind_device`。
- `user_id`：要绑定到哪个用户，默认当前用户。
- `device_name_hint`：可选。
- `expires_in_seconds`：默认 600。

输出字段：

- `bind_code`：只返回一次。
- `expires_at`。

#### 使用绑定码注册设备

`POST /api/v1/devices/register`

输入字段：

- `bind_code`。
- `device_name`。
- `platform`。
- `os_version`。
- `app_version`。
- `client_generated_device_id`：可选。

输出字段：

- `device_id`。
- `device_token`：只返回一次。
- `permissions`。
- `user`。

#### 获取设备列表

`GET /api/v1/devices`

权限：

- 主用户或有管理权限的设备。

输出：

- 当前空间下所有用户与设备。
- 设备权限。
- 在线状态。
- 最近活跃时间。

#### 更新设备权限

`PATCH /api/v1/devices/{device_id}/permissions`

权限：

- `can_manage_space`。

可更新：

- 自动上传。
- 手动上传。
- 自动接收。
- 手动下载。
- 接收范围。
- 来源设备规则。

服务端必须记录 audit log。

#### 撤销设备

`POST /api/v1/devices/{device_id}/revoke`

效果：

- 设置 `revoked_at`。
- 该设备所有 token 立即失效。
- 关闭该设备现有 WebSocket 连接。
- 不再为该设备生成新投递。

#### 上传图片

`POST /api/v1/images`

Content-Type：

- `multipart/form-data`。

Headers：

- `Authorization`。
- `Idempotency-Key`：推荐，防止重复提交。

表单字段：

- `file`：图片文件。
- `sha256`：客户端计算的文件哈希。
- `source_kind`：`screenshot`、`manual_share`、`selected_album`、`unknown`。
- `source_display_name`：原始显示名。
- `source_media_id_hash`：安卓本地媒体 ID 的 hash，可选。
- `captured_at`：客户端认为图片创建时间，可选。
- `origin_image_id`：通常为空；如果来自服务器下载，必须填写。

服务端行为：

1. 校验设备未撤销。
2. 校验设备是否具备上传权限。
   - 自动上传必须检查 `can_auto_upload`。
   - 手动上传必须检查 `can_manual_upload`。
3. 校验文件类型为图片。
4. 校验文件大小不超过配置上限。
5. 计算服务端 sha256，并与客户端 sha256 比对。
6. 去重。
7. 保存图片。
8. 写入 images 表。
9. 根据权限生成 deliveries。
10. 通过 WebSocket 通知在线目标设备。

输出：

- `image_id`。
- `deduplicated`：是否命中重复。
- `created_deliveries_count`。
- `expires_at`。

#### 获取待接收投递

`GET /api/v1/deliveries/pending`

用途：

- 客户端启动或 WebSocket 重连后补收。

输出：

- 当前设备未完成、未过期、可下载的投递列表。

#### 下载图片

`GET /api/v1/images/{image_id}/download`

权限：

- 当前设备必须是该图片的合法目标设备，或者具备手动下载权限。

输出：

- 图片二进制。

服务端必须避免：

- 未授权设备通过猜 ID 下载图片。
- 被撤销设备下载。
- 跨主用户空间下载。

#### 投递 ACK

`POST /api/v1/deliveries/{delivery_id}/ack`

输入字段：

- `status`：`downloaded`、`failed`、`skipped`。
- `error_message`：失败时可填。
- `local_path_hint`：可选，仅用于调试，不应暴露敏感路径。

服务端行为：

- 更新 deliveries 状态。
- 记录审计或调试日志。

### 8.4 WebSocket 协议

地址：

- `wss://<server>/api/v1/ws`

鉴权：

- 优先使用 `Authorization` header。
- 如果客户端库不方便设置 header，可以使用短期 WS token，不要长期 token 放 URL。

客户端连接后发送：

```json
{
  "type": "hello",
  "device_id": "device_123",
  "app_version": "1.0.0",
  "platform": "linux"
}
```

服务器返回：

```json
{
  "type": "hello.ack",
  "server_time": "2026-06-17T12:00:00Z"
}
```

新图片事件：

```json
{
  "type": "image.created",
  "event_id": "evt_01...",
  "delivery_id": "del_01...",
  "image": {
    "id": "img_01...",
    "mime_type": "image/png",
    "file_size": 1048576,
    "width": 2400,
    "height": 1800,
    "sha256": "..."
  },
  "source": {
    "upload_user_id": "usr_...",
    "upload_device_id": "dev_...",
    "upload_device_name": "OnePlus Pad"
  },
  "created_at": "2026-06-17T12:00:00Z",
  "expires_at": "2026-07-17T12:00:00Z"
}
```

客户端收到后：

1. 检查本地设置是否仍允许自动接收。
2. 下载图片。
3. 保存到本地目录。
4. 如果启用，复制到剪贴板。
5. 调用 ACK。

心跳：

- 客户端每 30 秒 ping。
- 服务器超时 90 秒未收到心跳则断开。
- 客户端断开后指数退避重连，最大间隔 60 秒。

### 8.5 存储策略

MVP 可使用服务器本地磁盘：

- 存储根目录由环境变量配置，例如 `/var/lib/studyshot/storage`。
- 文件路径按日期和 hash 分层：
  - `images/2026/06/17/<image_id>_<sha256_prefix>.png`

不要把图片直接存入数据库。

后续可以替换为 S3 兼容存储：

- MinIO。
- Cloudflare R2。
- AWS S3。
- 其他兼容服务。

### 8.6 图片保留策略

默认建议：

- 图片保留 30 天。
- 可配置为 1 天、7 天、30 天、永久。
- 每个主用户空间可配置最大存储容量。
- 超过容量后按最早过期或最早上传清理。

清理任务：

- 每小时或每天运行。
- 删除过期图片文件。
- 标记 images.deleted_at。
- 将相关 pending deliveries 标记为 expired。

### 8.7 文件大小和类型限制

默认限制：

- 单张图片最大 30MB。
- 允许 MIME：`image/png`、`image/jpeg`、`image/webp`。
- 默认不允许视频。

原因：

- 学习截图通常约 1MB 到 10MB。
- GPT 识图需要清晰度，不应过度压缩。

安卓端可提供可选压缩：

- 默认上传原图。
- 可选“节省流量模式”：长边限制为 2560px，JPEG 质量 90。
- 节省流量模式不能作为默认，避免影响手写内容识别。

### 8.8 安全要求

必须：

- HTTPS/WSS。
- token 鉴权。
- 服务端权限校验。
- 设备撤销。
- 文件下载鉴权。
- 上传大小限制。
- MIME 嗅探，不只相信客户端文件名。
- 防路径穿越。
- 审计日志。

推荐：

- 登录密码使用 Argon2id 或 bcrypt。
- token hash 存储。
- 限速：
  - 登录接口限速。
  - 绑定码验证限速。
  - 上传接口按设备限速。
- 失败日志不要输出完整 token。

暂不强制：

- 端到端加密。

说明：

- 用户使用自己的服务器，MVP 可以先依赖 TLS 和服务端访问控制。
- 如果未来加入端到端加密，需要重新设计多设备密钥分发和剪贴板流程，不建议 MVP 同时做。

## 9. 安卓 App 详细需求

### 9.1 安卓 App 职责

安卓 App 既能上传也能接收。

必须支持：

- 登录或绑定设备。
- 查看当前设备权限。
- 配置自动上传开关。
- 配置自动上传来源。
- 监听截图新增。
- 自动上传新截图。
- 上传失败重试。
- 手动选择图片上传。
- 通过系统分享菜单接收图片并上传。
- 可选自动接收图片。
- 查看最近上传/下载记录。
- 显示同步状态和错误。

### 9.2 权限申请

Android 13+：

- 申请 `READ_MEDIA_IMAGES`。

Android 12 及以下：

- 申请 `READ_EXTERNAL_STORAGE`。

Android 14+：

- 系统可能提供“允许访问所有照片”或“选择照片和视频”。
- 如果用户选择部分照片访问，App 要提示：
  - 自动监听新增截图需要完整图片访问。
  - 部分访问模式下只能手动分享或手动选择上传。

不要默认申请：

- `MANAGE_EXTERNAL_STORAGE`。

Foreground Service：

- 实时监听和上传使用 data sync 类型。
- 需要常驻通知。
- 通知文案要清楚，例如“StudyShot 正在监听学习截图”。

电池优化：

- App 可以引导用户关闭电池优化或允许后台运行。
- App 不能假装一定可以绕过系统省电限制。
- 对一加/ColorOS/OxygenOS 设备，要在设置页显示“后台保活状态检查”。

### 9.3 自动上传模式

安卓端提供三种模式：

#### 实时学习模式

目标：

- 截图后尽快上传。

行为：

- 开启 Foreground Service。
- 注册 ContentObserver 监听 MediaStore 图片变化。
- 发现新图片后入队上传。
- 使用 WorkManager 负责实际上传和重试。

用户可接受：

- 状态栏有常驻通知。

#### 省电模式

目标：

- 降低后台常驻成本，但允许较高延迟。

行为：

- 使用 WorkManager Content URI trigger 或周期任务。
- 不保证 1 到 3 秒实时。

#### 手动模式

目标：

- 不后台监听。

行为：

- 用户通过分享菜单或 App 内选择图片上传。

默认建议：

- 首次安装后默认手动模式。
- 用户明确开启“实时学习模式”后才开始自动监听。

### 9.4 截图识别规则

默认只上传截图。

识别截图时不能只依赖文件名，因为不同系统路径不同。

推荐判断：

- MediaStore `RELATIVE_PATH` 包含：
  - `Screenshots`
  - `Screenshot`
  - `Pictures/Screenshots`
  - `DCIM/Screenshots`
- 或设备系统截图目录被用户手动选择。
- 或文件名包含常见截图前缀：
  - `Screenshot`
  - `截屏`
  - `屏幕截图`

实现要求：

- 允许用户在设置中选择“截图目录”或“指定相册”。
- 第一次检测到疑似截图目录时，可以提示用户确认。
- 不能默认扫描并上传所有图片。

### 9.5 MediaStore 监听细节

安卓系统可能多次触发同一张图片变化。

客户端必须：

- 记录上次处理的 MediaStore ID 或时间戳。
- 对候选图片计算 sha256。
- 本地维护已上传哈希表。
- 如果图片处于写入中，等待稳定后再上传。

判断图片写入完成：

- Android 10+ 可检查 `IS_PENDING`，必须等 `IS_PENDING = 0`。
- 文件大小连续两次查询一致后再读。
- 如果读取失败，延迟重试。

不要：

- 每次变化都全量扫描整个相册。
- 用绝对文件路径直接读图。Android 10+ 分区存储下很多场景不可用。
- 忽略重复事件。

### 9.6 上传队列

本地 Room 需要保存上传队列：

- `local_task_id`。
- `media_uri`。
- `source_kind`。
- `sha256`。
- `status`：`pending`、`uploading`、`uploaded`、`failed`、`skipped`。
- `retry_count`。
- `last_error`。
- `created_at`。
- `updated_at`。

上传失败策略：

- 网络断开：等待网络恢复。
- 服务器 5xx：指数退避重试。
- 401/403：停止重试，提示权限或登录问题。
- 文件不存在：标记失败。
- 文件重复：标记 uploaded 或 skipped，不反复上传。

### 9.7 安卓自动接收

安卓端也可以作为接收设备，但默认不建议强制开启。

如果开启自动接收：

- 使用 WebSocket 保持连接。
- 收到事件后下载。
- 默认保存到 App 私有目录：
  - `Android/data/<package>/files/received`
- 该目录添加 `.nomedia`。
- 不写入系统相册，避免触发自身监听。

如果用户选择“保存到相册”：

- 必须二次确认。
- 保存后把该图片 sha256 写入“已接收哈希表”。
- 自动上传逻辑遇到同哈希必须跳过。

### 9.8 安卓界面需求

主要页面：

1. 连接状态页。
   - 服务器地址。
   - 当前账号。
   - 当前设备名。
   - 设备权限摘要。
   - WebSocket 状态。

2. 上传设置页。
   - 自动上传开关。
   - 模式：实时学习、省电、手动。
   - 来源范围：截图、指定相册、手动分享。
   - 仅 Wi-Fi 上传开关。
   - 节省流量压缩开关。

3. 接收设置页。
   - 自动接收开关。
   - 接收来源范围。
   - 保存位置。
   - 是否通知。

4. 记录页。
   - 最近上传。
   - 最近下载。
   - 失败原因。
   - 手动重试。

5. 管理页。
   - 仅主用户或管理设备显示。
   - 用户列表。
   - 设备列表。
   - 权限配置。
   - 创建绑定码。
   - 撤销设备。

界面原则：

- 不要像网盘一样突出文件管理。
- 重点是“当前是否能自动快传”。
- 错误信息要可行动，例如“没有完整图片访问权限，无法自动监听截图”。

## 10. 电脑客户端详细需求

### 10.1 电脑客户端职责

电脑客户端同时支持：

- 登录或绑定设备。
- 保持 WebSocket 连接。
- 自动接收授权图片。
- 自动保存到固定目录。
- 可选自动写入系统剪贴板。
- 查看最近接收记录。
- 管理权限和设备。
- 开机自启。
- 托盘运行。

### 10.2 平台支持

MVP 必须支持：

- Windows 10/11。
- Ubuntu 桌面环境。

Linux 注意：

- X11 与 Wayland 剪贴板行为可能不同。
- 剪贴板失败不能影响图片下载。
- 必须有明确状态提示。

### 10.3 自动接收流程

1. 客户端启动。
2. 读取本地配置和 token。
3. 连接 WebSocket。
4. 拉取 pending deliveries；如有离线积压，弹窗询问用户接收或忽略。
5. 用户确认接收后才分批下载；在线新投递仍自动下载。
6. 保存到下载目录。
7. 如果启用“自动复制到剪贴板”，写入剪贴板。
8. ACK 服务端。

下载目录默认：

- Windows：用户图片目录下的 `StudyShot Relay`。
- Linux：`~/Pictures/StudyShot Relay`。

文件命名：

- `<source-device-name>_YYYYMMDD-HHMMSS.png`（UTC，同秒冲突自动加序号）
- 设备名要清理非法文件名字符。

### 10.4 剪贴板策略

默认建议：

- 开启“收到后复制最新图片到剪贴板”。

行为：

- 收到一张图片后复制该图片。
- 如果短时间收到多张，最新一张覆盖剪贴板。
- 不要自动切换窗口。
- 不要自动操作 GPT 页面。

失败处理：

- 剪贴板写入失败时，仍保存文件。
- 在客户端记录页显示失败。
- 可选显示系统通知：“图片已保存，但复制到剪贴板失败”。

### 10.5 桌面界面需求

建议使用托盘 + 小窗口。

页面：

1. 状态页。
   - 服务器连接状态。
   - 当前设备名。
   - 自动接收状态。
   - 最近一次接收时间。

2. 接收设置。
   - 自动接收开关。
   - 下载目录选择。
   - 收到后复制到剪贴板。
   - 收到后系统通知。
   - 开机自启。

3. 上传设置。
   - MVP 可只支持手动上传。
   - 不建议桌面端默认监听文件夹上传。

4. 记录页。
   - 最近接收图片。
   - 打开文件所在目录。
   - 复制到剪贴板。
   - 重试失败下载。

5. 管理页。
   - 主用户管理用户、设备、权限、绑定码。

布局与安全提示：

- 左侧导航固定，鼠标滚轮只滚动右侧内容区。
- 宽屏或全屏窗口中，右侧内容列在内容区内居中显示，避免内容贴左造成大面积单侧留白。
- 已绑定设备如果迁移自未确认的非本机明文 HTTP 配置，桌面端必须显示确认提示并阻止携带 token 的请求。
- 用户确认继续使用明文 HTTP 后才持久化授权；用户选择暂不连接时，本次会话内隐藏提示但继续阻止携带 token 的请求。
- Android 默认拒绝明文 HTTP（`network_security_config.xml` 将 `cleartextTrafficPermitted` 默认置为 `false`）；仅 `localhost` / `127.0.0.1` / `::1` 例外，用于本机测试。私网内 IP HTTP 部署需在 NSC 中显式加白名单或改用 HTTPS。

### 10.6 桌面本地数据

本地至少保存：

- 服务器地址。
- 设备 token。
- 设备 ID。
- 下载目录。
- 是否自动接收。
- 是否自动复制到剪贴板。
- 已下载 delivery ID。
- 最近日志。

本地日志：

- 控制大小，例如最多 10MB。
- 不记录完整 token。
- 不记录图片内容。

## 11. 管理功能

管理功能要集成在安卓 App 和电脑客户端中。

MVP 可以让管理功能先在电脑端完整，安卓端提供基础管理；但接口必须支持两端共用。

### 11.1 主用户管理能力

主用户可以：

- 查看所有用户。
- 创建子用户。
- 禁用子用户。
- 创建用户组。
- 把子用户加入或移出组。
- 查看所有设备。
- 修改设备名。
- 修改设备权限。
- 创建设备绑定码。
- 撤销设备。
- 查看最近上传记录。
- 查看审计日志。
- 设置图片保留时间。

### 11.2 子用户能力

子用户可以：

- 查看自己的设备。
- 修改自己设备显示名。
- 查看自己设备权限。
- 在权限允许时上传或接收。

子用户默认不能：

- 查看其他子用户图片。
- 修改权限。
- 创建设备绑定码。
- 撤销别人的设备。

除非主用户显式授权。

### 11.3 权限管理界面避免误解

界面上不要只显示“允许下载”。

要区分：

- 自动接收。
- 手动下载。
- 接收来源。
- 上传权限。

例子：

- “一加平板”：自动上传开启；自动接收关闭。
- “Ubuntu 笔记本”：自动上传关闭；自动接收开启；收到后复制到剪贴板。
- “安卓手机”：手动上传开启；自动接收关闭。

## 12. 关键业务流程

### 12.1 平板截图到电脑 GPT

前置：

- 平板已绑定。
- 平板 `can_auto_upload = true`。
- 平板开启实时学习模式。
- 电脑已绑定。
- 电脑 `can_auto_receive = true`。
- 电脑接收来源包含该平板。

流程：

1. 用户在平板笔记软件中截图。
2. Android 系统写入截图到 MediaStore。
3. 平板 App 的 ContentObserver 收到变化。
4. 平板 App 查询最新图片。
5. 平板 App 判断该图片属于截图来源。
6. 平板 App 等待图片写入完成。
7. 平板 App 计算 sha256。
8. 平板 App 检查本地是否已上传或已接收过同哈希。
9. 平板 App 入队上传。
10. WorkManager 执行上传。
11. 服务器鉴权并保存图片。
12. 服务器根据权限生成投递，排除平板自身。
13. 服务器通过 WebSocket 通知 Ubuntu 客户端。
14. Ubuntu 客户端下载图片。
15. Ubuntu 客户端保存到固定目录。
16. Ubuntu 客户端复制图片到剪贴板。
17. 用户在 GPT 页面按 Ctrl+V。

### 12.2 手机手动上传到电脑

前置：

- 手机 `can_manual_upload = true`。
- 电脑 `can_auto_receive = true`。

流程：

1. 用户在相册中分享图片到 StudyShot。
2. 安卓 App 接收分享 URI。
3. 用户确认上传。
4. App 上传。
5. 服务器分发给电脑。
6. 电脑自动下载并复制到剪贴板。

### 12.3 设备离线后补收

1. 电脑关机。
2. 平板上传多张截图。
3. 服务器为电脑生成 pending deliveries。
4. 电脑开机。
5. 客户端连接服务器。
6. 客户端调用 pending 接口。
7. 客户端按时间顺序下载未完成图片。
8. 客户端 ACK。

### 12.4 撤销设备

1. 主用户在管理页撤销某手机。
2. 服务器设置 `revoked_at`。
3. 服务器关闭该设备 WebSocket。
4. 该设备后续请求返回 401/403。
5. 该设备不会再收到投递。

## 13. 部署方案

### 13.1 服务器环境

用户已有香港服务器。

推荐部署：

- Docker Compose。
- 后端服务容器。
- PostgreSQL 容器。
- Caddy 或 Nginx 容器。
- 图片存储挂载到宿主机目录。

推荐外部访问：

- 必须使用域名和 HTTPS。
- 端口只暴露 443。
- 数据库端口不对公网开放。

如果没有域名：

- 可以临时用 IP + 自签证书测试。
- 但安卓和桌面处理自签证书麻烦，不建议作为长期方案。

### 13.2 环境变量

后端至少需要：

- `DATABASE_URL`
- `STORAGE_DIR`
- `PUBLIC_BASE_URL`
- `JWT_SECRET` 或 token 签名密钥
- `INITIAL_OWNER_LOGIN`
- `INITIAL_OWNER_PASSWORD`
- `MAX_IMAGE_SIZE_MB`
- `DEFAULT_RETENTION_DAYS`
- `CORS_ALLOWED_ORIGINS`，如果有管理网页才需要

### 13.3 备份

至少备份：

- PostgreSQL 数据库。
- 图片存储目录。
- 服务端配置。

个人使用可接受：

- 每日数据库 dump。
- 图片目录按需备份。

如果图片只保留短期：

- 可以只备份数据库和配置，不备份过期图片。

## 14. 非功能需求

### 14.1 性能

单用户场景目标：

- 1 到 5 台设备。
- 每分钟 0 到 10 张截图。
- 单张图片 1MB 到 10MB。
- 服务器 CPU 和内存占用低。

后端不需要复杂优化，但必须：

- 上传流式处理，不要把大文件反复完整读入内存。
- 下载支持流式响应。
- WebSocket 连接数量少，但要自动清理断线连接。

### 14.2 可靠性

必须：

- 上传任务可重试。
- 下载任务可重试。
- WebSocket 断线自动重连。
- 服务端重启后客户端能恢复。
- 数据库迁移可重复执行。

### 14.3 可观测性

后端日志：

- 登录成功/失败。
- 设备绑定。
- 设备撤销。
- 上传成功/失败。
- 投递生成。
- 下载成功/失败。
- 权限拒绝。

客户端日志：

- 连接状态。
- 上传队列状态。
- 下载状态。
- 权限缺失。
- 剪贴板错误。

日志注意：

- 不记录 token。
- 不记录完整图片内容。
- 不在普通日志中输出敏感路径。

## 15. 测试与验收标准

### 15.1 后端测试

必须测试：

- 未登录不能上传。
- 被撤销设备不能上传。
- 无上传权限设备上传返回 403。
- 无接收权限设备不会生成投递。
- 上传设备不会收到自己的投递。
- 相同 sha256 重复上传不会重复生成大量投递。
- 跨主用户空间不能下载图片。
- 图片过期后不能下载。
- 设备撤销后 WebSocket 断开。

### 15.2 安卓测试

必须测试：

- 授权完整图片访问后，截图可自动上传。
- Android 14 部分照片访问时，自动监听不可用并显示明确提示。
- 同一张截图触发多次 MediaStore 事件时，只上传一次。
- 图片写入未完成时不会上传半张或损坏文件。
- 网络断开后上传进入队列，恢复后自动上传。
- 实时学习模式有常驻通知。
- 关闭实时模式后不再后台监听。
- 下载图片保存到 App 私有目录后，不会被再次自动上传。
- 手动分享图片到 App 可以上传。

### 15.3 桌面测试

必须测试：

- Windows 可自动接收。
- Ubuntu 可自动接收。
- 下载目录可配置。
- 收到后自动复制到剪贴板。
- 剪贴板失败时文件仍保存。
- WebSocket 断线后自动重连。
- 客户端关闭期间产生的投递，重启后可以补收。

### 15.4 端到端测试

必须通过：

1. 平板截图后，Ubuntu 5 秒内收到并写入剪贴板。
2. 用户在 GPT 页面 Ctrl+V 可以粘贴图片。
3. 平板不会下载自己上传的图片。
4. 手机手动上传后，电脑可以自动接收。
5. 禁用电脑自动接收后，电脑不再自动下载。
6. 撤销平板后，平板无法继续上传。
7. 服务器重启后，客户端自动恢复连接。

## 16. 容易被后续 AI 写错的点

以下约束必须明确告诉编码 AI：

1. 不要做成普通网盘。
   - 本项目核心是低延迟截图快传。
   - 不需要复杂文件夹同步。

2. 不要默认上传整个相册。
   - 默认只监听截图。
   - 全相册上传必须手动开启并二次确认。

3. 不要让上传设备自动下载自己上传的图片。
   - 服务端生成投递时排除上传设备。
   - 客户端也要防御。

4. 不要把安卓下载目录放进相册并再次监听。
   - 默认用 App 私有目录。
   - 加 `.nomedia`。
   - 用 sha256 跳过已接收图片。

5. 不要只做轮询。
   - WebSocket 是主实时通道。
   - 轮询只用于补偿。

6. 不要依赖厂商云、微信、网盘、Firebase 作为核心链路。
   - 用户已有自己的服务器。

7. 不要把权限只设计到用户级。
   - 必须设备级权限。

8. 不要忽略 Android 14 部分照片权限。
   - 部分权限下不能承诺自动监听全部截图。

9. 不要申请过高权限。
   - 默认不使用 `MANAGE_EXTERNAL_STORAGE`。

10. 不要在 MVP 中自动操作 GPT 页面。
    - 电脑端只需要下载和写剪贴板。
    - 用户自己在 GPT 页面粘贴。

11. 不要把 token 写进日志。

12. 不要硬编码服务器地址、账号或路径。

13. 不要忽略离线补收。
    - WebSocket 消息可能错过，所以必须有 pending deliveries 接口。

14. 不要为了省事让服务器公开图片 URL。
    - 下载必须鉴权。

15. 不要过度压缩图片。
    - 手写内容需要清晰。

## 18. 推荐开发顺序

虽然本文档不是代码计划，但为避免后续 AI 乱做，这里给出推荐实现顺序：

1. 后端数据模型、鉴权、设备绑定。
2. 后端图片上传、下载、投递记录。
3. 后端 WebSocket 通知。
4. 桌面客户端接收、下载、剪贴板。
5. 安卓 App 手动上传。
6. 安卓 App 截图监听和自动上传。
7. 权限管理 UI。
8. 离线补收、重试、清理任务。
9. 端到端验收和打包。

原因：

- 先让“服务器到电脑接收”稳定，再处理安卓后台复杂性。
- 安卓自动监听是最大风险点，应该在基础链路可用后单独验证。

## 19. 未来增强

可选增强：

- 局域网直连模式：同 Wi-Fi 下绕过服务器，服务器只做发现和鉴权。
- 端到端加密：服务器无法看到图片内容。
- OCR：服务器或客户端识别手写/印刷文字。
- GPT API 集成：自动把图片发送到指定模型。
- 浏览器扩展：在 ChatGPT 网页中直接插入最新图片。
- 更轻量桌面客户端：Tauri 或原生 Rust/Go。
- 图片批量合并：多张连续截图合并为一次投递。
- 临时会话：学习一门课时创建 session，截图自动归类。

## 20. 最终验收描述

当系统完成后，用户应能这样使用：

1. 打开平板笔记软件写作业。
2. 电脑停留在 GPT 页面。
3. 平板截图。
4. 几秒后电脑端自动收到图片，并把图片放进剪贴板。
5. 用户在 GPT 页面按 Ctrl+V。
6. GPT 收到手写截图。
7. 整个过程不需要打开微信、不需要打开网盘、不需要切换平板 App、不需要等待厂商云同步。

这就是本项目的核心价值。所有实现细节都应服务于这个体验。
