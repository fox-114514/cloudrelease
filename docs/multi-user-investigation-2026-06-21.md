# StudyShot Relay 多用户管理调查报告

调查日期：2026-06-21
调查范围：后端数据模型、用户/设备鉴权、图片上传与投递、绑定码、Android/桌面/Linux 客户端及现有测试。

## 1. 结论摘要

当前项目具备“一个 owner 空间下挂多个 child 用户”的后端基础，但还不是成熟的多用户自助系统。准确地说，它是“空间级共享 + 设备级授权”，不是“每个用户拥有天然隔离的私人图库”。

核心结论：

1. **不同 owner 空间之间没有发现可直接利用的串图路径。** 图片上传、列表、下载、删除和自动投递的主要查询都带有 `ownerUserId` 或由目标设备/投递记录间接限定；现有测试也覆盖了跨 owner 下载和删除返回 404。
2. **同一 owner 空间内会不会串图，取决于接收设备的 `autoReceiveScope`。** `all_authorized_sources` 明确会接收空间内其他用户设备上传的图片；`same_user_only` 才是按用户隔离；`selected_devices` 可以精确控制，但目前没有第一方 UI 配置来源设备规则。
3. **图片不会在数据库里随机归错用户。** 服务端从设备 token 推导 `ownerUserId`、`uploadUserId`、`uploadDeviceId`，客户端不能在上传请求中指定这些字段。但如果设备绑定时选错目标用户，之后所有上传都会稳定地记到错误用户下。
4. **子用户绑定明显不方便，而且存在误绑 owner 的高概率路径。** 后端 Web 管理页可以选择目标用户；Android 和桌面端的“登录后生成绑定码”不传 `userId`，所以 owner 登录生成的码只能绑定 owner 自己。child 用户 JWT 又被后端禁止创建绑定码，因此 child 无法靠自己的账号完成绑定。
5. **child 用户权限很弱。** child JWT 目前基本只能登录和查看自己名下的设备；不能创建绑定码、修改自己的设备、撤销自己的设备、管理图片或用户。真正的动作权限都附着在设备上。
6. **`canManageSpace` 是近似 owner 的高危权限。** 无论设备属于 owner 还是 child，只要开启此权限，就能查看/删除全空间图片、管理所有设备权限、创建/禁用 child 用户、查看审计日志，并可继续给其他设备授予管理权。
7. **Group 当前只是管理元数据。** `Group` / `GroupMember` 没有接入图片可见性、投递生成或权限判断，不能用于防串图。

综合评价：

| 维度 | 评价 | 说明 |
|---|---|---|
| 跨 owner 空间隔离 | 较好 | 应用层关键路径均有空间过滤，现有测试覆盖部分越权场景 |
| 同空间用户隐私隔离 | 不足 | 设计上允许共享，图库和高权限下载是空间级，不是用户级 |
| 默认安全性 | 尚可 | 新设备默认不自动接收、不自动上传，仅允许手动上传 |
| 子用户自助能力 | 很弱 | 无法用 child JWT 自助绑定、改名、撤销或管理自己的设备 |
| 绑定易用性 | 较差 | 只有后端 Web 管理页完整支持“选择目标用户” |
| 客户端配置一致性 | 不足 | 多端 UI 能力不一致，设备服务端权限与本地开关缺少同步接口 |

## 2. 实际权限模型

### 2.1 三层身份

- `owner` 用户 token：登录后获得 7 天 JWT，可管理整个 `ownerUserId` 空间。
- `child` 用户 token：可登录，但管理能力非常有限。
- 设备 token：绑定后生成的长期随机 token，实际上传、下载、WebSocket 和大部分运行时能力由设备权限决定。

设备属于某个 `userId`，同时冗余保存所属 `ownerUserId`。图片也同时记录：

- `ownerUserId`：租户/空间边界；
- `uploadUserId`：上传设备属于哪个用户；
- `uploadDeviceId`：具体上传设备。

这个三元归属设计能够保留准确来源，也让 `same_user_only` 可以按上传用户过滤。

### 2.2 权限矩阵

| 操作 | owner JWT | child JWT | 普通设备 token | `canCreateInvite` 设备 | `canManageSpace` 设备 |
|---|---:|---:|---:|---:|---:|
| 登录 | 是 | 是 | 不适用 | 不适用 | 不适用 |
| 查看全空间设备 | 是 | 否 | 否 | 否 | 是 |
| 查看自己的设备 | 是 | 是 | 否 | 否 | 是 |
| 给自己创建绑定码 | 是 | **否** | 否 | 是 | 是 |
| 给其他用户创建绑定码 | 是 | 否 | 否 | 否 | 是 |
| 自动上传 | 否 | 否 | 按权限 | 按权限 | 按权限 |
| 手动上传 | 否 | 否 | 按权限 | 按权限 | 按权限 |
| 自动接收 | 否 | 否 | 按权限和范围 | 按权限和范围 | 按权限和范围 |
| 查看/删除全空间图库 | 是 | 否 | 否 | 否 | 是 |
| 管理用户/组/审计 | 是 | 否 | 否 | 否 | 是 |
| 修改任意设备权限 | 是 | 否 | 否 | 否 | 是 |

需要强调：权限单位是设备，不是用户。owner 名下设备默认也没有管理权；反过来，child 名下设备一旦获授 `canManageSpace`，就具有空间级管理能力。

## 3. 图片是否会串

### 3.1 跨 owner 空间

正常 API 路径下不会：

- 上传时的空间和用户来自已认证设备，不信任客户端自报身份；
- 图片下载先用调用方 `ownerUserId` 查找图片；
- 管理图库和删除图片按 `ownerUserId` 过滤；
- 自动投递只查找与图片相同 `ownerUserId` 的目标设备；
- 绑定码选定的目标用户必须属于同一个 owner 空间；
- 接收来源规则创建时验证目标和来源设备都属于当前空间。

物理文件没有按用户或 owner 单独建目录，而是统一存放在 `images/YYYY/MM/DD/`。不过文件名包含服务端生成的 UUID 和哈希前缀，数据库 `storageKey` 唯一；文件读取只能经由先鉴权、再查数据库的下载接口。因此这是“逻辑隔离、共享物理存储”，不是物理租户目录隔离。

防御纵深仍有不足：多个表的 `ownerUserId` 只是普通字符串，没有外键或复合外键约束来证明关联记录属于同一空间。当前安全性主要依赖每条应用查询写对过滤条件；未来新增接口时容易漏掉。

### 3.2 同一 owner 空间内

这里是明确可能“串”的地方，而且部分是产品设计：

| `autoReceiveScope` | 实际行为 | 多用户建议 |
|---|---|---|
| `disabled` | 不生成自动投递 | 最安全，适合只上传设备 |
| `all_authorized_sources` | 接收同空间所有授权上传设备的图片，排除上传设备自身 | **会跨用户，不适合需要隐私隔离的 child** |
| `same_user_only` | 仅当目标设备 `userId == uploadUserId` 时投递 | 多用户默认推荐 |
| `selected_devices` | 仅接收显式启用的来源设备 | 最精确，但当前 UI 不完整 |

另外还有三个空间级可见性事实：

1. owner JWT 和任何 `canManageSpace` 设备都能查看、预览、删除全空间所有用户的图片。
2. `canManualDownload` 不是“下载自己图片”的权限，而是“只要知道 image ID，就能下载同一空间内任意未过期图片”的权限。普通设备没有全图库列表入口，所以直接利用需要先得到 image ID，但权限边界仍是空间级。
3. 管理图库返回上传用户和上传设备信息，但没有按用户筛选参数；owner 看到的是合并图库。

因此，如果产品目标是“家庭空间内 owner 可以监管、child 之间互相不可见”，当前模型只能部分满足：自动投递可用 `same_user_only` 隔开，但图库、手动下载和管理权限仍以空间为边界。

### 3.3 图片归属是否会错

上传接口不接受 `uploadUserId` 或 `ownerUserId`，所以普通客户端不能伪造上传者。真正的风险发生在绑定阶段：

1. 绑定码记录 `targetUserId`；
2. 注册设备时设备永久关联到该用户；
3. 后续上传全部使用设备记录中的 `userId`；
4. `same_user_only` 也使用这个 `userId` 判断。

如果 child 的设备误用 owner 绑定码：

- 图片会被标记为 owner 上传；
- owner 的 `same_user_only` 接收设备可能收到这些图片；
- child 自己的 `same_user_only` 接收设备反而收不到；
- 当前没有“迁移设备到另一用户”的接口，只能撤销后重新绑定。

这不是随机串图，而是绑定目标错误导致的确定性错误归属，用户感知上会非常像串图。

### 3.4 去重不会把两个用户合并

一小时去重条件包括 `ownerUserId + uploadDeviceId + sha256`。相同图片由不同设备或不同用户上传时会创建不同图片记录，不会因为哈希相同被合并到另一个用户。但这也意味着同一图片由多个设备上传时可能产生重复投递。

## 4. 设备绑定体验调查

### 4.1 后端支持的正确流程

理想流程是：

1. owner 创建 child 用户；
2. owner 创建绑定码时明确选择该 child 的 `userId`；
3. child 设备用一次性码注册；
4. owner 为设备开启所需上传/接收权限；
5. 接收设备设置为 `same_user_only` 或配置具体来源设备。

设备注册默认权限为：

```json
{
  "canAutoUpload": false,
  "canManualUpload": true,
  "canAutoReceive": false,
  "canManualDownload": false,
  "canManageSpace": false,
  "canCreateInvite": false,
  "autoUploadScope": "screenshot_only",
  "autoReceiveScope": "disabled"
}
```

这个默认值不会自动串图，但注册完成后仍需要 owner 二次配置，否则自动上传和自动接收都不会工作。

### 4.2 各客户端实际能力

| 入口 | 创建 child | 生成码时选择目标用户 | 修改接收范围 | 配置 `selected_devices` 来源 | 结论 |
|---|---:|---:|---:|---:|---|
| 后端 `/admin` Web 页 | 是 | 是 | 是 | 否 | 当前最完整，但仍缺来源规则 UI |
| Android 管理页 | 否 | 否 | **只显示，不能修改** | 否 | 无法独立完成 child 的正确绑定和接收配置 |
| 桌面 Electron 管理页 | 否 | 否 | 是 | 否 | 可管设备权限，但 owner 登录生成码只会绑定 owner |
| Linux CLI/Web | 否 | 否，主要输入已有绑定码 | 本地运行开关为主 | 否 | 依赖后端 Web 管理页 |
| 直接 API | 是 | 是 | 是 | 是 | 能力完整，但不适合普通用户 |

### 4.3 具体不便和缺口

1. **child JWT 不能创建自己的绑定码。** 后端注释和实现都要求 child 通过一个已授权设备行动；但第一方客户端没有用当前设备 token 创建绑定码的完整入口。
2. **Android/桌面快捷生成码默认绑定登录者。** 请求不传 `userId`，owner 登录时目标就是 owner；界面没有醒目标注“此码将绑定到谁”。
3. **Android child 登录后的 UI 与权限不匹配。** child 可以登录并看到自己的设备，但页面仍显示“生成绑定码”、权限开关、撤销/删除等操作；点击后由服务端返回 403。
4. **Android 不能修改接收范围。** 它只能切换 `canAutoReceive`，但新设备范围仍是 `disabled`；即使打开自动接收，也不会产生投递。
5. **`selected_devices` 在 UI 中是半成品。** Web/桌面允许选择这个枚举，却没有界面选择来源设备。选择后规则表为空，结果是完全收不到图。
6. **服务器权限与客户端本地开关不同步。** 注册响应带回权限，但 Android 没有把这些服务端权限作为持续状态保存；各客户端也缺少 `/devices/me` 一类接口刷新自己的有效权限。因此用户可能在本地打开自动上传/接收，实际请求却持续被服务器拒绝或永远没有投递。
7. **无法重新归属设备。** 误绑后不能由 owner 修正 `userId`，只能撤销、删除显示记录、重新生成正确目标用户的码再绑定。
8. **用户生命周期不完整。** owner 可以创建、重命名、禁用 child，但不能重置密码，也没有删除 child 的接口。
9. **`invite_child_user` 未完成。** 枚举和绑定码 API 接受该 purpose，但设备注册明确拒绝它，项目中也没有消费该邀请的用户注册流程。

## 5. 风险分级

### 高优先级

#### H1. child 设备容易误绑到 owner

影响：图片归属错误、`same_user_only` 路由错误、审计来源错误。
原因：只有后端 Web 管理页能在生成码时选择 child；Android/桌面快捷流程不支持目标用户。
建议：所有生成码 UI 必须显示并确认目标用户；管理者可选 child；child JWT 应允许只给自己生成设备码，或为已有设备提供 `canCreateInvite` 自助入口。

#### H2. 同空间隐私边界容易被 `all_authorized_sources` 打开

影响：某个 child 的图片自动投递到另一个用户设备。
原因：该范围的语义就是全空间；UI 使用英文枚举，缺少跨用户隐私警告。
建议：有多个用户时默认推荐或强制 `same_user_only`；选择全空间来源时明确显示“会接收其他用户上传的图片”并二次确认。

#### H3. `canManageSpace` 权限过宽且可自我扩散

影响：持有该设备 token 的人可读取/删除全空间图片、管理所有用户和设备，并给其他设备授予同等权限。
建议：拆分为设备管理、用户管理、内容管理、审计查看等权限；至少禁止设备 token 修改自身/其他设备的 `canManageSpace`，该权限只允许 owner JWT 授予。

### 中优先级

#### M1. `selected_devices` 无可用 UI

影响：用户选择后完全收不到图片，难以定位原因。
建议：在所有暴露该选项的管理端同时实现来源设备复选列表；未配置来源时显示阻断性提示。

#### M2. child 自助管理几乎不可用

影响：每次绑定、改名、撤销或权限调整都依赖 owner。
建议：定义“仅管理自己设备”的 child 能力，包括创建自己的绑定码、重命名/撤销自己的设备、查看自己的有效权限；不能访问其他用户设备。

#### M3. 图库和手动下载缺少用户级边界

影响：一旦设备获得 `canManualDownload` 或 `canManageSpace`，边界就是整个空间。
建议：如果产品需要 child 隐私，增加 `own_user_only` 内容权限，默认下载条件为“有 delivery 或图片属于当前 user”；跨用户下载使用单独显式权限。

#### M4. Group 没有业务效果

影响：管理者以为分组可以控制共享范围，实际不会影响任何投递。
建议：要么把 Group 接入投递和图库授权，要么在功能完成前从 UI 隐藏并标注“仅组织标签”。

### 防御纵深与一致性问题

- 数据库没有用复合约束保证关联记录的 `ownerUserId` 一致；建议在服务层集中 tenant-scoped repository，并为关键关联增加数据库约束或触发器。
- 绑定码可为已禁用 child 创建，注册时也没有过滤 `disabledAt`；设备会注册成功但随后无法认证，体验混乱。
- `expiresInSeconds` 只有正数校验，没有最大值；高权限设备可以创建长期有效的一次性码。
- 管理 Web 页把 7 天 JWT 存入 `localStorage`，与“管理 token 不应长期保存”的权限文档原则不完全一致。

## 6. 推荐产品模型

在继续开发前应明确选择以下一种模型：

### 模型 A：家庭共享空间

- owner 和被授权管理设备能看全部图片；
- child 之间默认 `same_user_only`；
- owner 可显式开启跨用户接收；
- UI 明确说明 owner 可监管全部内容。

这与当前后端最接近，改造成本较低。

### 模型 B：严格用户私有空间

- 图片列表/下载默认按 `uploadUserId` 隔离；
- owner 是否可看 child 内容需要单独监管权限或审计同意；
- Group 作为显式共享单位；
- `canManageSpace` 不再自动等于内容读取权。

这需要调整权限语义和多条 API，不应只靠修改接收范围实现。

基于项目目前定位，建议采用模型 A，但把“child 默认隔离、owner 可监管、跨用户共享需显式开启”写进产品界面和测试。

## 7. 推荐落地顺序

### 第一阶段：先防误绑和误收

1. Android、桌面端生成绑定码时增加目标用户选择和醒目确认。
2. 允许 child JWT 仅为自己创建 `bind_device` 绑定码；或者实现已有设备通过 `canCreateInvite` 创建自己的码。
3. 新增 `GET /devices/me`，返回当前设备所属用户和实时权限；客户端绑定确认页显示“将绑定到：用户 X”。
4. 多用户空间中，接收设备默认使用 `same_user_only`；对 `all_authorized_sources` 加跨用户警告。
5. 为“child A 上传、A 接收、B 不接收、owner 是否接收”增加集成测试。

### 第二阶段：补齐配置闭环

1. 实现 `selected_devices` 来源规则 UI。
2. Android 支持修改服务端接收范围，而不只是显示。
3. child UI 根据角色隐藏无权限操作，并提供自己可执行的设备动作。
4. 增加误绑后的安全修复流程；若不允许迁移设备，至少提供“一键撤销并为正确用户重新生成码”。

### 第三阶段：拆分高危权限

1. 拆分 `canManageSpace`；只有 owner JWT 能授予最高权限。
2. 把内容读取权与设备/用户管理权分离。
3. 决定 Group 是共享权限单元还是纯标签，并让 UI 与行为一致。
4. 完成密码重置、用户删除/归档和 `invite_child_user`，或删除未实现的接口语义。

## 8. 当前可用的安全配置

如果现在就要部署多用户，建议：

- child 上传设备：`canAutoUpload=true`（如需要）、`canManualUpload=true`、`canManageSpace=false`、`canManualDownload=false`。
- child 接收设备：`canAutoReceive=true`、`autoReceiveScope=same_user_only`、`canManageSpace=false`。
- owner 管理设备：只在可信主机上开启 `canManageSpace`，移动设备尽量不开。
- 不要给 child 接收设备使用 `all_authorized_sources`。
- 暂时不要在 UI 选择 `selected_devices`，除非同时通过 API 配好 `receive_source_rules`。
- 只从后端 `/admin` 页面为 child 生成绑定码，并在生成时核对目标用户；不要使用 Android/桌面端 owner 快捷生成码给 child 绑定。
