# StudyShot Relay 多用户体验优化方案

版本：V2 提案
日期：2026-06-21
依据：[多用户管理调查报告](./multi-user-investigation-2026-06-21.md)

详细实现接口、逐文件任务和验收标准见：[多用户 V2 设计](../../design/multi-user-v2.md)。如两份文档存在差异，以详细开发规格为准。

## 1. 要解决的问题

当前实现让用户同时理解 owner、child、用户 JWT、设备 token、六个布尔权限、上传范围、接收范围和来源设备规则。后端模型能工作，但正常操作需要 owner 在多个管理页之间来回切换，任何一步选错都会表现为“绑定失败”“收不到图”或“图片串了”。

V2 的目标不是增加更多开关，而是让以下三件事自然成立：

1. 普通成员可以自己绑定自己的第二台设备，不需要每次找 owner。
2. 新设备默认只接收同一用户的图片，除非用户明确开启共享。
3. 大多数用户只选择“这台设备拿来做什么”，不用理解底层权限字段。

建议产品定位采用“家庭/团队共享空间”模型：owner 负责空间治理并可查看全空间内容，普通成员之间默认隔离，跨成员共享必须显式开启。这个规则必须在创建空间和邀请成员时说清楚。

## 2. 用户能理解的心智模型

界面只暴露三个主要对象：

- **空间**：例如“我的家庭”或“高三一班”。
- **成员**：owner 在界面显示为“空间管理员”，child 改称“成员”。数据库枚举可以暂时不改。
- **设备**：明确显示“属于谁”和“用途”。

底层仍可保留设备 token 和细粒度权限，但默认页面不直接展示技术枚举。

### 2.1 设备用途预设

绑定时选择一个用途：

| 预设 | 用户看到的说明 | 底层权限 |
|---|---|---|
| 只上传截图 | 把这台设备的截图发到自己的其他设备 | 自动/手动上传；不自动接收 |
| 只接收我的图片 | 接收我名下其他设备上传的图片 | 自动接收；`same_user_only` |
| 我的设备双向同步 | 既上传，也接收我自己的图片 | 上传 + 自动接收；`same_user_only` |
| 接收指定设备 | 只接收勾选来源的图片 | 自动接收；`selected_devices` |
| 接收空间全部图片 | 会收到其他成员的图片 | `all_authorized_sources`，二次确认 |
| 管理终端 | 管理空间，不建议用于普通移动设备 | 单独的委派管理机制，不再等同普通设备权限 |

“高级设置”页面才展示具体权限字段，且每个字段使用中文描述和风险提示。

## 3. 新的绑定流程

### 3.1 成员绑定自己的设备

首选流程改为登录绑定：

1. 输入服务器地址。
2. 输入成员账号和密码。
3. 服务端返回空间、成员名称和角色。
4. 页面明确显示“这台设备将属于：张三”。
5. 选择设备用途预设。
6. 确认后注册设备并保存设备 token；用户 JWT 立即丢弃。

成员不再需要先登录后台生成一段绑定码，再回到绑定页粘贴。

安全边界：登录绑定只能把设备绑定给当前登录用户，不能指定其他 `userId`。

### 3.2 已绑定设备帮助绑定第二台设备

提供“添加我的设备”：

1. 已绑定设备生成 10 分钟有效的二维码/短码。
2. 码的目标用户由服务端锁定为当前设备所属用户。
3. 新设备扫码后显示空间和成员名称，再确认用途。
4. 使用一次后立即失效。

底层可以复用 `canCreateInvite`，但普通用户不需要看到这个权限名。

### 3.3 owner 代成员绑定设备

用于儿童、教室平板或没有独立密码的托管成员：

1. owner 在“成员 → 张三 → 添加设备”中生成二维码。
2. 二维码页面始终显示目标成员头像/名称。
3. 新设备确认页再次显示目标成员。
4. 注册成功后自动回到该成员的设备列表。

禁止提供没有目标成员提示的“快速生成绑定码”。这正是当前最容易误绑 owner 的入口。

### 3.4 误绑修复

增加“转移设备”流程，而不是要求用户手工撤销、删除、重绑：

1. 只有 owner 可以发起转移。
2. 转移会立即吊销旧设备 token。
3. 服务端生成一个锁定新成员的短期重新认领码。
4. 设备输入/扫描后获得新 token，并从此把新上传归到新成员。
5. 历史图片保留原上传用户，不批量改写审计历史。

## 4. 权限模型优化

### 4.1 用户身份负责“能管理什么”

建议用户角色简化为：

- 空间管理员：管理成员、设备、共享策略和审计。
- 成员：管理自己的设备和自己的图片。
- 托管成员：不能登录，由管理员代管设备。

第一阶段可以继续使用数据库中的 `owner` / `child`，只在 API 能力和 UI 文案上实现；后续再迁移枚举。

成员 JWT 应允许：

- 给自己绑定设备；
- 查看、改名、撤销自己的设备；
- 查看自己的图片；
- 修改自己设备的非高危用途设置；
- 不能查看其他成员设备或授予管理权限。

### 4.2 设备能力负责“能传、能收什么”

普通设备只保留运行时能力：

- 上传；
- 接收；
- 本地手动操作。

`canManageSpace` 不应继续作为可由另一台设备随意切换的普通权限。建议：

1. 只有 owner JWT 可以授予或撤销委派管理。
2. 委派管理记录独立保存，包含授予人、过期时间和最近使用时间。
3. 委派管理默认 30 天过期，可手工续期。
4. 委派管理员不能给其他设备继续授予最高权限。

这样设备 token 泄露时不会永久接管整个空间。

### 4.3 图片访问负责“能看谁的内容”

把内容权限从 `canManageSpace` 和 `canManualDownload` 中拆开：

- 成员默认只能列出、预览、下载自己上传的图片，以及投递给自己的图片。
- owner 可以查看全空间，但图库默认先显示“我的图片”，另有“全部成员”筛选。
- 跨成员手动下载需要显式的共享记录，不能只凭知道 image ID。
- `canManualDownload` 改成“允许在当前可见范围内手动下载”，不再自动扩大可见范围。

## 5. 图片共享和接收规则

### 5.1 默认规则

- 新设备默认不跨成员接收。
- 开启自动接收时，默认同时设置 `autoReceiveScope=same_user_only`。
- `canAutoReceive=true + autoReceiveScope=disabled` 这种无效组合不再允许从普通 UI 产生。
- 选择 `all_authorized_sources` 时显示成员名单和隐私警告，并要求二次确认。

### 5.2 把 Group 变成真正的共享组

当前 Group 没有业务效果。建议只保留一种明确用途：共享范围。

例如创建“父母”“数学小组”，然后配置：

- 哪些上传成员/设备是来源；
- 哪些成员/设备可以接收；
- 是否允许查看历史图库；
- 是否只允许自动投递、不允许浏览历史。

如果短期内不实现这些规则，应从 UI 隐藏 Group，避免用户误以为分组已经产生权限隔离。

### 5.3 来源设备选择必须形成闭环

当用户选择“接收指定设备”时，同一页面立即展示来源设备复选框。没有选择任何来源时：

- 保存按钮禁用；
- 显示“至少选择一个来源设备”；
- 不允许静默保存一个永远收不到图的配置。

## 6. API 调整建议

### 第一阶段：兼容现有数据模型

新增或调整：

```text
GET    /api/v1/me
GET    /api/v1/devices/me
POST   /api/v1/devices/register-with-login
POST   /api/v1/bind-codes              child 可为自己创建
GET    /api/v1/users/me/devices
PATCH  /api/v1/users/me/devices/:id    改名/用途预设
POST   /api/v1/users/me/devices/:id/revoke
GET    /api/v1/users/me/images
POST   /api/v1/devices/:id/transfer    owner 发起重新认领
```

关键约束：

- child JWT 创建绑定码时服务端强制 `targetUserId=request.user.userId`，忽略客户端传入的其他用户。
- 所有绑定/注册响应返回 `spaceName`、`targetUserId`、`targetDisplayName`，客户端必须展示。
- `GET /devices/me` 返回服务端实时权限和所属用户，客户端每次启动及收到 403 后刷新。
- 用途预设在服务端原子更新多个权限字段，避免客户端只更新 `canAutoReceive` 却忘记接收范围。

### 第二阶段：规范化空间模型

如果未来支持一个账号加入多个空间，建议从 `ownerUserId` 自引用升级为：

```text
Space
  id
  name

Membership
  spaceId
  userId
  role
  status

Device
  membershipId

Image
  spaceId
  uploadMembershipId
  uploadDeviceId
```

这能用外键明确空间归属，减少每张表手工维护 `ownerUserId` 导致的越权风险。当前只有单 owner 空间时不必阻塞第一阶段体验优化。

## 7. 各客户端统一要求

Android、Electron、Linux Web 必须共享同一套行为：

- 绑定前显示目标空间和成员；
- 绑定时选择用途预设；
- 首页显示“当前身份：张三 · 我的设备”；
- 服务端权限变化后自动刷新；
- 无权限功能直接隐藏或解释，不显示一个必然返回 403 的按钮；
- 接收范围使用中文：关闭、仅我的设备、指定设备、空间全部；
- 所有跨成员操作显示来源成员和目标成员。

后端 `/admin` 保留高级管理能力，但不应成为普通成员完成日常绑定的必经入口。

## 8. 分阶段落地计划

### Sprint 1：消除最反人类的路径

1. child 允许给自己创建绑定码。
2. Android/桌面端绑定码页面显示目标成员；owner 可选择成员。
3. 增加 `GET /devices/me`，客户端显示当前所属成员和实时权限。
4. 普通 UI 使用设备用途预设，自动接收默认 `same_user_only`。
5. child UI 隐藏空间级管理按钮，开放自己设备的改名和撤销。
6. 补两个 child 用户之间不串图的集成测试。

验收标准：一个新成员只拿到账号密码，就能独立绑定两台设备并互传图片，不需要 owner 操作，也不会收到其他成员图片。

### Sprint 2：共享配置闭环

1. 实现“指定来源设备”复选 UI。
2. Group 接入共享规则，或暂时移除 Group UI。
3. 成员增加自己的图库；owner 图库增加成员筛选。
4. 手动下载改为“可见范围内下载”。
5. 实现设备转移/重新认领。

验收标准：用户可以在一个页面回答“这台设备要接收谁的图”，并立即得到与界面一致的结果。

### Sprint 3：高危权限治理

1. 拆除普通设备权限中的 `canManageSpace` 自扩散能力。
2. 增加有期限、不可转授的委派管理会话。
3. 增加密码重置、成员归档/删除、邀请状态管理。
4. 完成或删除 `invite_child_user` 半成品流程。
5. 为 `ownerUserId` 关联增加数据库防御约束。

验收标准：泄露一个普通设备 token 最多影响该设备可见的图片和操作，不能接管整个空间。

## 9. 推荐立即执行的最小改造

如果只安排一个开发周期，优先做以下六项：

1. child JWT 可以为自己生成绑定码。
2. 所有绑定确认页显示目标成员，禁止无提示绑定。
3. 新设备选择“接收”用途时自动使用 `same_user_only`。
4. 增加 `/devices/me` 并在客户端展示有效权限。
5. Android 可以真正修改接收范围；选择指定设备时必须选择来源。
6. `canManageSpace` 只能由 owner JWT 授予。

这六项不需要立即重做数据库，却能消除目前最主要的误绑、收不到图、跨用户串图和权限扩散问题。
