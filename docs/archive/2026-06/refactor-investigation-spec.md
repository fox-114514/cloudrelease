# StudyShot Relay 重构调研与新项目规格文档

生成日期：2026-06-25  
调研范围：项目需求文档、协议文档、后端、Android、桌面端、Linux CLI/Web、部署与发布文档、测试与已知问题记录  
定位：本文用于后续从零重构或新项目重建。它不要求沿用现有技术栈、目录结构或代码组织，只抽象需要解决的问题、必须具备的功能、交付产出、风险与解决方案。

---

## 1. 一句话结论

这个项目要解决的核心问题是：把安卓平板或手机上的学习截图，以私有、低延迟、低打断的方式自动送到电脑或其他授权设备，并尽可能自动放入剪贴板，让用户在 GPT、Claude、编辑器或任意目标窗口里直接粘贴。

它不是网盘、相册同步、聊天软件，也不是 AI 客户端。它的产品价值来自一条极短链路：

```text
截图 -> 自动上传到私有中继 -> 授权设备实时接收 -> 保存并可选写入剪贴板 -> Ctrl+V
```

新项目必须保留这条链路的低摩擦体验、安全边界和离线补偿能力。技术栈可以全部替换，但协议契约、权限模型、端到端可靠性和防循环规则不能随意弱化。

---

## 2. 项目解决的问题

### 2.1 原始场景

用户在安卓平板上手写笔记、做题或推导，同时在电脑或手机上使用 GPT/Claude 等工具辅助学习。用户经常需要把平板上的手写内容截图发到电脑端的 AI 对话框或其他应用。

已有方式的问题：

- 平板分屏：发热、卡顿、影响手写体验。
- 微信文件助手：步骤多，需要频繁切换应用和下载文件。
- 厂商跨设备中转：依赖品牌生态，跨品牌不可用，延迟经常达到几十秒到一分钟。

真正的问题不是“能不能传文件”，而是“如何在不打断学习思路的情况下，把刚截的图快速出现在另一个设备的剪贴板里”。

### 2.2 产品目标

新项目应实现：

- 安卓设备截图后自动上传。
- 授权接收设备几秒内收到图片。
- 电脑端自动保存，并可选自动写入系统剪贴板。
- 离线设备重连后能看到待补收图片数量，由用户决定接收或忽略。
- 多用户、多设备、设备级权限可控。
- 所有图片下载必须鉴权，不暴露公开 URL。
- 不依赖微信、厂商云服务或第三方网盘。

### 2.3 明确不做

新项目不应扩张成：

- 完整网盘。
- 完整相册备份。
- 聊天系统。
- 图片编辑器。
- 默认上传全相册的同步工具。
- 默认接入 GPT/OpenAI API 的客户端。
- 必须发布到应用商店的商业化 App。

## 3. 成功标准

### 3.1 体验指标

- 正常网络下，截图完成到电脑收到：理想 1-3 秒，可接受 5 秒内。
- 弱网或远端服务器：10 秒内可接受，但客户端必须显示状态。
- 接收端自动保存到固定目录。
- 接收端可选自动写入剪贴板。
- 用户保持在目标窗口，无需切回传输 App。
- Android 实时模式允许常驻通知，但不能明显影响手写体验。

### 3.2 稳定性指标

- 常规截图不能漏传。
- MediaStore 多次事件不能导致重复上传。
- 网络失败要自动重试。
- 服务端重启后客户端自动重连。
- 离线期间产生的投递能在重连后补收。
- 被撤销设备不能继续上传、下载或接收事件。
- 不能因为下载目录被监听而产生上传-下载循环。

### 3.3 安全指标

- 外网生产环境必须使用 HTTPS/WSS。
- 非回环 HTTP 必须由用户显式确认，并展示持续风险提示。
- 设备绑定后才能使用。
- token 不进入普通日志、崩溃日志或 UI 截图。
- 服务端 token 不明文存储。
- 客户端 token 尽量使用系统安全存储；安全存储不可用时不得静默降级写明文。
- 权限以设备为最小执行单位。
- 所有租户资源必须通过 owner/space 边界隔离。
- 关键操作保留审计日志。

## 4. 核心业务对象

新项目可以重新命名和建模，但必须保留以下概念。

### 4.1 空间

一个私有中继空间，属于一个空间管理员。空间是数据隔离边界。不同空间之间不能共享设备、用户、图片、投递和审计日志。

现项目使用 `ownerUserId` 作为空间边界。新项目可以引入显式 `Space` 表或等价模型，但必须让所有查询先限定空间。

### 4.2 用户

至少两类：

- 空间管理员：管理整个空间、成员、设备、权限、图片和审计。
- 成员：管理自己的设备和自己的图片，默认不能跨成员访问。

新项目可以把 `child` 文案统一为“成员”。成员默认只在自己的设备之间同步，跨成员接收必须显式配置。

### 4.3 设备

设备是权限执行的最小单位。每台 Android、Windows、Linux 电脑都应独立绑定、独立撤销、独立授权。

设备至少需要记录：

- 设备 ID。
- 所属空间。
- 所属用户。
- 设备名。
- 平台。
- App 版本。
- 系统版本。
- token hash。
- 创建时间。
- 最后在线时间。
- 撤销时间。
- 软删除时间。
- 设备权限。

### 4.4 图片

每次上传产生一个图片对象。图片至少需要记录：

- 图片 ID。
- 所属空间。
- 上传用户。
- 上传设备。
- SHA-256。
- MIME。
- 文件大小。
- 宽高。
- 存储位置。
- 来源类型：截图、手动分享、选定相册、未知。
- 原始来源显示名。
- 创建时间。
- 过期时间。
- 删除时间。

### 4.5 投递

图片上传后，服务端根据权限为目标设备生成投递记录。投递记录是离线补偿和 ACK 的基础。

状态至少包括：

- pending：待通知或待补收。
- notified：已通过 WebSocket 通知。
- downloaded：目标设备已下载。
- failed：目标设备处理失败。
- skipped：用户或系统跳过。
- expired：图片过期或被删除。

## 5. 权限模型

### 5.1 权限原则

- 用户 token 用于管理和手动操作。
- 设备 token 用于自动上传、自动接收、下载、ACK。
- 设备权限是服务端最终事实。
- 客户端本地开关只能控制“是否启动本地任务”，不能绕过服务端权限。
- owner 用户 token 才能授予高危权限。
- owner 名下设备不能因为“属于 owner”就自动获得管理权。
- 成员默认不能跨成员访问图片、设备或投递。

### 5.2 设备权限字段

新项目至少应支持以下能力：

| 权限 | 含义 |
|---|---|
| canAutoUpload | 允许自动上传监听到的图片 |
| canManualUpload | 允许手动上传或分享上传 |
| canAutoReceive | 允许服务端为该设备生成自动投递 |
| canManualDownload | 允许手动查看或下载历史图片，但不得跨成员 |
| canManageSpace | 允许管理空间内设备、图片、成员等运行时配置 |
| canCreateInvite | 允许为自己或授权目标创建设备绑定码 |

### 5.3 上传范围

自动上传范围至少应支持：

- screenshot_only：默认，仅截图。
- selected_album：用户选择的相册或目录，可配置排除子目录。
- manual_share_only：不自动监听，仅手动分享。
- all_images：高风险，不推荐，必须二次确认。

默认必须是 `screenshot_only` 或更保守的手动模式。不能默认扫描或上传整个相册。

### 5.4 接收范围

自动接收范围至少应支持：

- disabled：不自动接收。
- same_user_only：仅同一用户名下设备。
- selected_devices：只接收显式来源设备。
- all_authorized_sources：接收空间内全部授权来源，必须仅管理员或管理设备可配置。

成员默认应为 `same_user_only` 或 `disabled`，不能默认跨成员。

### 5.5 用途预设

为了避免 UI 直接暴露多个布尔权限，新项目应保留“用途预设”抽象：

| 预设 | 自动上传 | 手动上传 | 自动接收 | 上传范围 | 接收范围 |
|---|---:|---:|---:|---|---|
| manual_only | 否 | 是 | 否 | 仅手动 | 不接收 |
| upload_only | 是 | 是 | 否 | 仅截图 | 不接收 |
| receive_own | 否 | 是 | 是 | 仅手动 | 仅我的设备 |
| sync_own | 是 | 是 | 是 | 仅截图 | 仅我的设备 |
| custom | 服务端推断 | 服务端推断 | 服务端推断 | 服务端推断 | 服务端推断 |

`custom` 只能由服务端根据当前权限推断，客户端不能提交。

## 6. 核心用户流程

### 6.1 首次部署

1. 用户准备一台 Linux 服务器。
2. 部署后端、数据库和 HTTPS 反向代理。
3. 初始化空间管理员账号。
4. 浏览器进入管理后台。
5. 创建成员或直接使用 owner 空间。

产出：

- 可访问的 HTTPS API。
- 可访问的 Web 管理后台。
- 可用于客户端绑定的服务器地址。
- 初始管理员凭据。

### 6.2 设备绑定

推荐流程：

1. 客户端填写服务器地址。
2. 用户输入绑定码，或登录账号自助绑定。
3. 客户端先调用绑定码预览，展示目标空间和目标成员。
4. 用户确认目标正确。
5. 客户端提交设备名、平台、系统版本、App 版本和用途预设。
6. 服务端签发设备 token。
7. 客户端安全保存 token 和设备身份。
8. 客户端立即刷新服务端真实权限。

关键要求：

- 绑定码一次性使用。
- 默认 10 分钟过期，最长不超过 1 小时。
- 绑定码大小写敏感，但输入时忽略首尾空白。
- 预览不消耗绑定码。
- 注册前必须让用户知道这台设备将绑定到哪个成员。
- 注册成功后不应保存临时登录 JWT。

### 6.3 Android 自动上传

1. 用户开启自动上传。
2. App 检查权限、设备 token、服务端权限、HTTPS/HTTP 策略。
3. 默认监听 MediaStore 图片变化。
4. 只查询最近新增图片，不扫描全相册。
5. 根据截图目录、文件名、选定相册和排除目录筛选候选图片。
6. 为候选图片创建本地上传任务。
7. 实时模式下前台服务直接上传，WorkManager 作为兜底。
8. 上传前计算 SHA-256 和 MIME。
9. 服务端校验 hash、权限、重复和循环风险。
10. 上传成功后记录本地已上传 hash。

关键要求：

- 不能在主线程做网络或 hash。
- MediaStore 多次回调必须合并。
- ColorOS/MIUI/HarmonyOS 等 ROM 可能延迟回调，实时模式需要轻量主动扫描最近媒体。
- 非实时模式可以只依赖可靠队列，延迟可略高。

### 6.4 服务端投递

1. 服务端收到图片并写入存储。
2. 服务端创建图片记录。
3. 服务端查找同空间、未撤销、未禁用用户、允许自动接收的目标设备。
4. 排除上传设备自己。
5. 根据目标设备接收范围判断是否应接收。
6. 为符合条件的设备生成投递记录。
7. 如果目标设备在线，通过 WebSocket 发送 `image.created`。
8. 投递记录保留 pending/notified 状态，供离线补偿。

关键要求：

- 不能因为 WebSocket 通知失败就丢投递。
- 投递生成必须在上传事务或等价一致性边界内完成。
- `same_user_only` 必须严格按上传用户和目标设备用户匹配。
- `selected_devices` 必须依赖显式规则。

### 6.5 接收端自动下载

1. 客户端用设备 token 建立 WebSocket。
2. 连接后发送 hello。
3. 服务端返回 hello.ack。
4. 客户端检查 pending 数量。
5. 新图片事件到达时，若本地开启自动接收且服务端允许自动接收，则下载图片。
6. 下载后校验 SHA-256。
7. 保存到目标目录。
8. 可选写入系统剪贴板。
9. 本地记录接收历史和 received hash。
10. 调用 ACK 标记 downloaded。

关键要求：

- 下载必须走鉴权接口。
- 下载到本地应使用唯一文件名，避免并发覆盖。
- 写文件失败或 hash 不匹配要清理半成品。
- 剪贴板失败不能导致投递失败，因为文件已经保存。
- ACK 失败不能导致重复下载，客户端应能幂等处理。

### 6.6 离线补偿

1. 设备离线期间，服务端继续保留 pending/notified 投递。
2. 设备重连后查询 pending。
3. 客户端展示待补收数量。
4. 用户选择接收或跳过。
5. 接收时逐批下载并 ACK。
6. 跳过时逐批 ACK skipped。

关键要求：

- pending API 要分页或限制批量，避免一次返回过多。
- 客户端要防止一个 delivery 在并发路径被重复处理。
- 用户可明确跳过旧投递，避免突然下载大量历史图片。

### 6.7 手动图库与下载

1. 有权限用户或设备可以列出图片。
2. owner 或管理设备可查看全空间。
3. 成员只看自己的图片。
4. 设备凭 canManualDownload 只看和下载自己用户的图片，或已有投递关系的图片。
5. 删除图片应软删除数据库记录并尽量删除磁盘文件。
6. 删除图片后 pending/notified 投递应变为 expired。

关键要求：

- 跨成员图片访问优先返回 404，避免枚举。
- 默认图片列表不显示软删除图片。
- 过期图片不能继续下载。
- 图片文件名建议包含来源设备名和上传时间。

### 6.8 设备管理

功能应包括：

- 设备列表。
- 设备改名。
- 用途预设切换。
- 自动接收范围配置。
- 权限查看。
- 撤销设备。
- 删除已撤销设备。
- 成员自助管理自己的设备。
- owner 管理全空间设备。

关键要求：

- 删除设备必须先撤销。
- 删除应为软删除，保留历史图片和审计。
- 删除设备时清理接收来源规则，并跳过其 pending 投递。
- 撤销设备时立即关闭 WebSocket。

## 7. 客户端能力清单

### 7.1 Android 客户端

必须能力：

- 设备绑定：绑定码绑定、账号自助绑定、绑定码预览。
- 身份显示：当前空间、成员、设备、用途预设、权限快照。
- 自动上传：截图监听、选定相册、排除目录、Wi-Fi only。
- 实时学习模式：前台服务、常驻通知、最近媒体轻量扫描、即时上传。
- 可靠队列：WorkManager 或等价机制，指数退避重试。
- 手动上传：文件选择或系统分享菜单。
- 自动接收：WebSocket、pending、下载、通知。
- 图片库：查看、预览、另存或系统分享。
- 权限同步：启动、定时、403/401、WebSocket 异常后刷新服务端权限。
- 安全存储：设备 token 加密保存；不可用时阻止绑定/上传/接收。
- HTTP 安全策略：默认 HTTPS，非回环 HTTP 显式确认。
- 帮助和诊断：权限、后台运行、厂商限制、数据位置、常见问题。

建议保留的技术行为：

- ContentObserver + 最近媒体查询。
- 实时模式每秒查询最近媒体作为 OEM 回调延迟兜底。
- 上传前创建可靠队列兜底任务。
- received_hashes 防止服务端下载图片再上传。
- manual upload 缓存文件上传成功或终态失败后清理。

### 7.2 桌面客户端

必须能力：

- 设备绑定：绑定码、账号自助绑定、绑定码预览。
- WebSocket 自动接收。
- pending 补收和跳过。
- 自动保存到配置目录。
- 可选写入系统剪贴板。
- 最近接收记录。
- 桌面通知。
- 托盘常驻、暂停/恢复。
- 目录监听自动上传。
- 排除下载目录和任意子目录，避免循环。
- 目录变更立即持久化。
- 服务端权限同步后自动停用无权限任务。
- 设备管理和图片库入口。
- 自托管更新检查、下载、SHA-256 校验和安装包拉起。

实现注意：

- 单文件上传失败不能让监听器状态变成停止。
- 底层 watcher 致命错误必须关闭旧 watcher 并允许重启。
- 设备 token 存储优先系统安全存储；安全存储不可用时至少使用权限受限配置文件，并给用户明确警告。
- 非回环 HTTP 未确认时，不能发起任何携带 token 的请求，也不能启动 watcher 或 WebSocket。

### 7.3 Linux CLI / Web 客户端

必须能力：

- CLI 绑定：绑定码预览确认、账号自助绑定、隐藏密码输入。
- CLI 身份查询：whoami、permissions、refresh-permissions。
- CLI 上传单图。
- CLI 目录监听上传。
- CLI WebSocket 接收。
- CLI run 模式同时运行接收和监听。
- 自动写入剪贴板：Wayland 优先 wl-copy，X11 使用 xclip，且只允许图片 MIME。
- 本地 Web 管理界面。
- 本地 Web 必须有会话鉴权和 Origin 校验，不能把 device token 暴露给浏览器。
- 本地 Web 配置 DTO 必须脱敏。
- 自托管更新检查与安装包校验。

实现注意：

- CLI 配置文件应原子写入。
- 长运行模式要定时刷新服务端权限。
- 下载目录和监听目录应做安全路径约束，允许用户显式覆盖但要告警。

- 更新元数据与安装包下载。
- 审计日志查询。

### 8.2 协议约定

建议保留统一响应 envelope：

```json
{
  "success": true,
  "data": {}
}
```

错误响应：

```json
{
  "success": false,
  "error": {
    "code": "ERROR_CODE",
    "message": "Human readable message",
    "details": {}
  }
}
```

客户端必须按 `error.code` 决策，不按 message 字符串拼接。

关键 HTTP 行为：

- 401：身份缺失、token 无效、设备撤销、用户禁用。客户端停止自动任务并提示重新绑定或登录。
- 403：身份有效但权限不足。不要无限重试同一操作。
- 400：请求格式或业务参数错误。客户端修正请求。
- 404：不存在、跨空间、跨成员、过期。避免泄露资源存在性。
- 409：状态冲突，如登录名已存在、设备未撤销不能删除。
- 429：限速，客户端退避。
- 5xx：服务端错误，客户端指数退避。

### 8.3 存储能力

图片存储要求：

- 服务端计算 SHA-256 并校验客户端传入值。
- 存储 key 不应可被猜测。
- 下载只通过鉴权接口。
- 删除图片时数据库软删除，磁盘文件 best-effort 删除。
- 存储根目录可配置。
- 默认保留期可配置。

数据库要求：

- 所有空间数据都有空间边界。
- 设备 token 存 hash。
- 绑定码存 hash。
- 关键字段建立索引：空间、用户、设备、过期时间、投递目标和状态、hash。
- 迁移可重复部署。

### 8.4 实时通道

WebSocket 要求：

- 只能使用设备 token。
- 用户 JWT 不能用于 WebSocket。
- 同一设备新连接建立时关闭旧连接，使用与鉴权失败不同的关闭码。
- hello / hello.ack。
- ping / pong。
- 心跳超时断开。
- 上传后在线目标收到 image.created。
- 客户端声明更新通道后可收到 app.update.available。
- WebSocket 通知失败不影响 pending 投递。

### 8.5 安全与运维

必须能力：

- 登录独立限流。
- 登录失败日志不能包含密码。
- 用户不存在时执行 dummy password 校验，减少时序差异。
- JWT 限定算法。
- CORS 可配置。
- 请求体大小限制与反向代理一致。
- SIGTERM/SIGINT 优雅停机：停止接收新请求、关闭 WebSocket、停止清理定时器、关闭数据库连接。
- 清理任务：过期图片、过期投递、存储清理。
- 部署脚本或文档明确 HTTPS 和纯 IP HTTP 的差异。

## 12. 可能遇到的问题与解决方案

### 12.1 Android 后台监听延迟

问题：

部分国产 ROM 会延迟 MediaStore ContentObserver 回调，导致后台截图上传不稳定。

解决：

- 实时学习模式使用前台服务。
- ContentObserver 与最近媒体主动扫描并行。
- 主动扫描只查最近短窗口，不扫全相册。
- 非实时模式接受较高延迟，依靠 WorkManager 兜底。

### 12.2 Android 15 前台服务限制

问题：

targetSdk 35 后，dataSync 前台服务可能受到 24 小时内总时长限制。

解决：

- 在 target 35 分支做实机验证。
- 实现服务超时回调并给用户明确提示。
- 将“实时学习模式”设计为用户主动开始/结束的学习会话。
- 超时后降级为 WorkManager 或较低频轮询。
- 不为绕过限制滥用 specialUse 前台服务类型。

### 12.3 误传全相册

问题：

如果默认监听范围过宽，可能上传隐私照片。

解决：

- 默认只截图。
- selected_album 需要用户明确选择。
- all_images 必须二次确认并持续显示风险。
- 服务端记录 sourceKind 和来源显示名，方便审计。
- 客户端 UI 明确显示当前自动上传范围。

### 12.4 上传-下载循环

问题：

接收端下载的图片被本地监听器再次上传，造成循环。

解决：

- 服务端不投递给上传设备自己。
- 接收端下载目录默认排除在监听目录外。
- 接收端记录 received_hashes，自动上传前跳过已接收 hash。
- 上传请求带 originImageId 时服务端拒绝。
- Android 保存到 gallery 时也要记录 received hash。

### 12.5 重复上传

问题：

同一张截图可能触发多次 MediaStore 或文件系统事件。

解决：

- 本地以 URI/mediaIdHash 或 sha256 去重。
- WorkManager/队列使用唯一任务名。
- 服务端对同一设备一小时内相同 sha256 去重。
- 客户端对已上传 hash 建 LRU 或持久缓存。

### 12.6 WebSocket 不可靠

问题：

网络切换、服务端重启或代理断开会丢实时事件。

解决：

- WebSocket 只用于通知。
- 投递记录作为事实来源。
- hello.ack 后查询 pending。
- 心跳超时重连。
- 指数退避和抖动，避免重连风暴。
- ACK 幂等。

### 12.7 权限撤销后客户端仍运行

问题：

本地开关可能还开着，服务端权限已关闭。

解决：

- 客户端启动时刷新 `/devices/me`。
- 定时刷新服务端权限。
- 上传/下载遇到 401/403 后停止对应自动任务。
- WebSocket 1008 后清除绑定或提示重绑。
- UI 显示“本地开关”和“服务端权限”两层状态。

### 12.8 跨成员隐私泄露

问题：

多用户场景下，成员可能通过 imageId 或设备 ID 探测他人资源。

解决：

- 所有资源查询先限定空间。
- 成员资源访问再限定 uploadUserId 或 device.userId。
- 跨成员访问返回 404 而不是 403。
- canManualDownload 不能跨成员下载已知 image ID。
- all_authorized_sources 只能管理员或管理设备配置。
- selected_devices 对成员只允许自己名下来源。

### 12.9 设备 token 泄露

问题：

设备 token 如果被日志、本地 Web 或明文存储泄露，攻击者可上传/下载。

解决：

- 服务端只存 token hash。
- 客户端优先系统安全存储。
- 安全存储不可用时阻止绑定，或至少明确警告且权限受限。
- 本地 Web 不返回 token。
- 本地 Web 使用 HttpOnly Cookie 和 Origin 校验。
- 日志中严禁输出 Authorization、token、完整请求体。
- 设备可单独撤销，撤销立即关闭 WebSocket。

### 12.10 非 HTTPS 使用风险

问题：

纯 IP HTTP 会明文传输 token、密码和图片。

解决：

- 默认补全 `https://`。
- 只有 loopback HTTP 默认允许。
- 非回环 HTTP 要显式开关或 CLI 参数。
- 配置中持久记录 `allowInsecureHttp`。
- 未确认旧配置不能发任何 token 请求。
- UI 保持风险横幅。
- 文档建议只在 VPN/局域网临时使用。

### 12.11 图片存储膨胀

问题：

截图高频上传会积累大量图片。

解决：

- 默认保留期。
- 过期清理任务。
- 图片库手动删除。
- 存储大小监控。
- 可选按空间配置保留策略。
- 删除时数据库软删除，文件 best-effort 删除并记录日志。

### 12.12 大图内存压力

问题：

移动端或 Web 后台预览大图时容易占内存。

解决：

- 服务端缩略图接口。
- 客户端采样解码。
- 流式上传/下载。
- 避免同时保存完整 ByteArray 和全尺寸 Bitmap。
- 下载写临时文件，校验后 rename。

### 12.13 发布和更新复杂

问题：

多端版本、签名、安装包、SHA-256 和服务端元数据容易不一致。

解决：

- 单一版本源。
- 发布脚本生成 SHA256SUMS。
- 后端更新元数据来自环境或发布表。
- Android versionCode 必须递增且使用同一签名。
- Windows/Linux 安装包下载后必须校验 hash。
- 客户端更新通道和平台严格匹配。

### 12.14 测试数据库和 CI

问题：

后端权限隔离依赖数据库，若本地没有 PostgreSQL 就容易跳过关键测试。

解决：

- CI 配置 PostgreSQL service。
- 后端集成测试自动迁移独立测试库。
- 无数据库单元测试只覆盖纯函数，不替代集成测试。
- 四端 build/typecheck 进入 CI。

## 15. 验收测试清单

### 15.1 主链路

- Android 绑定成功。
- 桌面端绑定成功。
- 桌面端开启自动接收和剪贴板。
- Android 截图后 5 秒内桌面保存文件。
- 桌面剪贴板可直接粘贴图片。
- 文件名包含来源设备和时间。
- 服务端有图片、投递和审计记录。

### 15.2 离线

- 接收端离线。
- 上传端上传多张截图。
- 接收端重连后显示 pending 数量。
- 选择接收后逐张下载并 ACK。
- 选择跳过后投递变 skipped。

### 15.3 权限

- 关闭上传设备 canAutoUpload 后自动上传失败且客户端停止。
- 关闭接收设备 canAutoReceive 后不生成新投递。
- 撤销设备后 WebSocket 关闭。
- 被撤销设备上传、下载、pending、ACK 均失败。
- 成员不能访问其他成员图片。
- owner 可访问全空间图片。

### 15.4 防循环与去重

- 上传设备不会收到自己上传的图。
- 接收目录在监听目录内时仍不会再次上传。
- 同一 MediaStore 事件多次触发只产生一条上传任务。
- 同一设备短时间重复上传相同 hash 被去重。

### 15.5 安全

- 无 token 下载图片返回 401。
- 错误 token 返回 401。
- 过期图片不能下载。
- 非回环 HTTP 未确认时客户端不发送 token。
- 本地 Web API 无 Cookie 返回 401。
- 本地 Web 配置响应不包含 deviceToken。
- 日志中搜索不到 token 和密码。

### 15.6 运维

- 数据库迁移可在空库执行。
- 服务端 SIGTERM 优雅退出。
- 清理任务不会阻塞进程退出。
- Caddy/反代请求体限制与后端一致。
- 更新包 SHA-256 错误时客户端拒绝安装。

## 16. 重构迁移建议

### 16.1 不建议直接“搬目录”

现有项目已经能运行，但多端代码耦合在仓库结构里。新项目如果为了彻底区分，应从领域和协议开始，而不是复制目录。

建议先固化：

- 领域模型。
- API 契约。
- WebSocket 契约。
- 错误码。
- 权限矩阵。
- 端到端验收脚本。

然后再选择技术栈。

### 16.2 保留现有测试思想

最有价值的不是具体测试框架，而是测试覆盖范围：

- 多用户隔离。
- 绑定码生命周期。
- 用途预设。
- 权限升级防护。
- 图片跨成员访问。
- 设备撤销。
- HTTP 安全策略。
- 本地 Web 鉴权。
- watcher 错误状态。

新项目应把这些变成契约测试或黑盒测试，防止换技术栈时功能回退。

### 16.3 兼容旧客户端的取舍

如果新项目完全替代旧项目，可以不兼容旧客户端。但如果要平滑迁移，需要：

- 保留 `/api/v1` 或提供兼容层。
- 允许旧设备重新绑定。
- 提供旧存储迁移或重新绑定说明。
- 图片历史可以迁移，也可以明确只迁移账号和设备，旧图片归档。

建议新项目早期不承诺无缝迁移所有历史图片，先确保主链路稳定。

### 16.4 数据迁移优先级

迁移优先级：

1. 用户/空间。
2. 设备与权限。
3. 绑定关系重新生成或重新绑定。
4. 图片元数据。
5. 图片文件。
6. 审计日志。

如果时间有限，设备重新绑定通常比迁移 token 更安全。

## 17. 关键开放问题

新项目立项前建议明确：

1. 是否仍然以自托管为唯一部署模式，还是提供局域网直连/内网中继。
2. 是否必须保留 Windows、Linux 桌面、Linux CLI/Web 三种接收端，还是合并为一个跨平台客户端。
3. 是否在第一版就支持多用户，还是先单用户闭环。
4. 是否需要图片历史长期保存，还是只保留短期中转。
5. 是否接受纯 IP HTTP，只用于 VPN/局域网并显式授权。
6. Android targetSdk 35 的前台服务策略何时验证。
7. 是否需要端到端加密。当前项目是传输加密和服务端鉴权，不是服务端不可见的 E2EE。
8. 是否需要多实例后端。如果需要，WebSocket 广播和 pending 通知要引入共享通道。

## 18. 推荐的新项目边界

为了避免新项目再次变成“什么都能传”的工具，建议写入产品原则：

- 默认只为学习截图服务。
- 自动上传默认只截图。
- 手动上传可以传任意图片，但必须显式操作。
- 所有跨成员行为默认关闭。
- 所有远端下载必须鉴权。
- 所有客户端都以服务端权限为准。
- 实时通道只是提示，投递记录才是可靠事实。
- 剪贴板是体验增强，不是投递成功的唯一标准。
- 安全存储失败时宁可不可用，也不静默泄露 token。

## 20. 最终建议

重构时不要从“用什么框架重写”开始，而应先固定三件事：

1. 截图到剪贴板的端到端体验指标。
2. 设备级权限、空间隔离和投递可靠性契约。
3. Android 后台上传与接收端下载/剪贴板的可验证实现。

只要这三件事保持清晰，新项目可以完全换技术栈、换目录、换 UI、换部署方式。反过来，如果这三件事没有先写成契约，即使复用现有代码，也很容易在多端重构中丢掉这个项目真正有价值的能力。
